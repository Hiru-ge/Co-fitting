### Issue #214: リロードを挟むと1日3件を超えて訪問できてしまうバグを修正する

提案を途中（例：1件訪問済み）でリロードすると新たな3件が提案され、結果として1日に3件以上の訪問が記録できてしまう。日次の訪問上限（3件）をサーバーサイドで強制する制限が欠落している。

根本原因: 提案カウント（Redis）と訪問カウント（DB / Redis）が独立して管理されており、リロードによって提案セットが再生成されると訪問機会が無制限に増える。

**修正方針:**
- `POST /api/visits` 時に当日の訪問件数を確認し、3件以上の場合は 429 を返す
- 訪問カウントは `visits` テーブルの当日レコード数を正とする（Redis補助可）
- リロード（`force_reload`）は提案のリセットのみ行い、訪問上限には影響しない

- [x] 🔴 RED: 当日の訪問が3件に達した状態で4件目の訪問記録を作成しようとすると 429 が返ることを確認するテストを書く
- [x] 🟢 GREEN: `POST /api/visits` に日次訪問カウントチェックを追加し、3件超で拒否するロジックを実装する
- [x] 🔵 REFACTOR: 日次訪問カウントの取得クエリを整理し、タイムゾーン（JST基準）を統一する

### Issue #184: 提案リロード機能の実装と設定変更の即時反映化

Issue #182（翌日反映通知）を廃止し、設定変更（興味タグ・提案半径）を即時反映する設計に変更する。ホーム画面にリロードボタンを追加し、リロードは1日3回まで制限する。残り回数はUIに表示する。

**設計方針:**
- 設定変更（興味タグ・提案半径）は即時反映。変更時に当日の提案キャッシュをクリアし、次のリロードで新設定が使われる
- `POST /api/suggestions` に `force_reload: true` フラグを追加。リロード時は日次リロードカウントを消費して新しい提案を生成する
- 通常の初回取得はリロードカウントを消費しない
- 提案APIレスポンスに `reload_count_remaining`（残りリロード回数）フィールドを追加
- 1日3回リロード後の上限到達時: 「今日のリロードは使い切りました。明日また使えます」を表示
- 1日のリロード回数は 0:00（JST）にリセット
- 将来拡張としてプレミアムプランでの上限引き上げを想定した設計にする（今は実装しない）
- **UI変更**: ホーム画面左下にある「次の提案カードに進む」ボタンは不要なため削除し、同位置にリロードボタン（+ 残り回数表示）を配置する

- [x] 🔴 RED: `force_reload: true` で3回超の提案リクエストが429を返すことを確認するテストを書く。また、興味タグ・提案半径変更後に次回リクエストで新設定が反映されることを確認するテストを書く
- [x] 🟢 GREEN: バックエンドにリロードカウント管理（Redisで日次リセット）を追加し、`force_reload` フラグのロジックを実装する。フロントにリロードボタン・残り回数表示（「今日あとN回リロードできます」）を追加する
- [x] 🔵 REFACTOR: リロードカウントのRedisキー設計を整理し、既存の日次提案カウント・提案キャッシュとの責務分離を確認する

### Issue #196: New Places API（v1）に移行して提案カバレッジを抜本改善する

**背景**: 旧 Places API の `NearbySearch` は最大 20 件・人気度順のため、小規模な地元施設がヒットしない（例: 近所のカフェが提案されない）。New API への移行で以下を同時解決する。

**解決する問題**:
- 論点1: 地元施設がAPIで拾えない（`rankPreference: DISTANCE` 距離順化で解決）
- 論点6: 6ジャンル（ラーメン・カラオケ・ゲームセンター・温泉・スポーツジム・海川湖）のマッピング欠落（New API タイプ対応）

**New API タイプ追加マッピング**:
| ジャンル | New API types |
|---|---|
| ラーメン・麺類 | `ramen_restaurant` |
| カラオケ | `karaoke` |
| ゲームセンター | `amusement_center`, `video_arcade` |
| 温泉・銭湯 | `public_bath`, `sauna` |
| スポーツジム | `gym`, `fitness_center` |
| 海・川・湖 | `beach`, `lake`, `river` |

**アーキテクチャ方針**:
- エンドポイント: `POST https://places.googleapis.com/v1/places:searchNearby`
- Go SDK（`googlemaps/google-maps-services-go`）は旧 API 専用のため、`net/http` で REST 直接呼び出しに変更
- `rankPreference: DISTANCE`（距離順）で地元施設を優先
- `includedTypes`: 全 visitableTypes を明示指定（不要な施設タイプを除外）
- `fieldMask`: `places.id,places.displayName,places.location,places.types,places.photos,places.rating`
- 料金: Enterprise SKU（rating 取得） = **$35/1,000**（旧 API の $40/1,000 より安い）
- Redis キャッシュ（24h TTL）は継続。Phase 1 ドッグフーディング段階なら無料枠（5,000 回/月）に十分収まる

**優先度**: 高 / **工数**: 大

- [x] 🔴 RED: New API クライアントのモックを作成し、距離順レスポンスが正しく変換されることを確認するテストを書く
- [x] 🟢 GREEN: `GooglePlacesClient.NearbySearch` を `net/http` ベースの New API 呼び出しに置き換える。`PlacesSearcher` インターフェースは維持してテストへの影響を最小化する
- [x] 🔵 REFACTOR: `placeTypeToGenreName` に 6 ジャンル × N タイプを追加。`visitableTypes` も New API タイプに合わせて更新。既存テストが全てパスすることを確認する
