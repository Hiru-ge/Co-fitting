## 論点 1: Google Maps API の施設カバレッジ

### 概要
実際に存在する施設のうち Google Places API で呼び出せるものがどれだけあるか。「Roamble で提案できる施設」＝「Places API に登録されている施設」なので、カバレッジが低ければプロダクト価値そのものに直結する。

### 調査すべきこと
- **都市部**（東京・大阪等）と**地方部**での施設数の差
- **ジャンル別のカバレッジ差**: 飲食（カフェ・レストラン）は豊富だが、ニッチなジャンル（ゲームセンター・温泉など）はどうか
- **APIの検索タイプ制限**: 現在 `visitableTypes`（28タイプ）でフィルタしているが、このリストの妥当性
- **チェーン店の割合**: 提案がコンビニ・チェーン飲食店だらけにならないか
- **営業時間データの精度**: Places APIで営業中かどうかの判定がどこまで正確か
- **実態調査**: 実際にユーザーの近所の施設をAPIで検索し、近くに存在するのに提案されない施設がどれくらいあるのか、ユーザーの体験に即した調査を行う

### 検証方法（案）
- 実際に数地点（都心・郊外・地方）で API を叩き、返却される施設数・種類を比較する
- 結果を「ジャンル × エリア」のマトリクスで整理する
- 開発者の現在地でAPIを叩いて周辺3kmの全施設を一旦リストアップし、実態との乖離を確認する
- ユーザー(開発者含む)からのフィードバックを集め、実際に「近所のあの店が提案されない」という声がどれくらいあるかを定量化する

## 論点 2: 仕様書と現コードの認識齟齬

### 概要
要件定義書（requirements.md）と実装コードの間にある乖離を特定・解消する。

### 調査済みの乖離一覧

| # | 項目 | 仕様書 | 実装 | 影響 |
|---|------|--------|------|------|
| A | レベルテーブル | Lv.50まで（指数カーブ） | **Lv.10まで（5,000XPで打ち止め）** | レベルキャップが低すぎ。継続利用で早期にカンストする |
| B | 称号 | 「旅立ちの冒険者」「街の探検家」等 | **未実装** | 表示されていない |
| C | ストリークボーナスXP | 「連続週数 × 10 XP」 | **未実装**（ストリークカウントは更新されるがXPボーナスは付与されない） | 長期継続のインセンティブが弱い |
| D | 初エリアボーナス | +30 XP | **常に `false` で呼び出されている**（バッジ判定は機能するがXPボーナスは事実上無効） | エリアパイオニアバッジは実装済みだがXPボーナスなし |
| E | 検索半径の選択肢 | 「1km / 3km / 5km / 10km」 | **3km〜30km のスライダー（デフォルト10km）** | 仕様書の記述が古い。実装の方が妥当 |
| F | 脱却頻度設定 | 「低・中・高」で調整 | **未実装**（固定で3件中1件が興味外） | ユーザーが脱却の頻度をコントロールできない |
| G | 位置情報の保存 | 「Place IDのみサーバーに保存（精密位置データは端末のみ）」 | **`visit_history` テーブルに `lat`, `lng` カラムが存在し保存されている** | プライバシーポリシーとの矛盾（→論点10） |
| H | 技術スタック | 「Go (net/http or Echo/Chi)」(CLAUDE.md) / 「Go (Gin)」(requirements.md) | **Echo を使用** | ドキュメント間の不一致のみ。実害なし |

### 対応方針の判断が必要な項目(ユーザーに必ず確認をとって方針を決定すること)
- A〜D: **仕様書を実装に寄せるか、実装を仕様書に寄せるか**（特にレベルテーブルは重要）
- D: **初エリア**のような曖昧な用語の認識を詰め、搭載するバッジの命名や内容を改めて明確にする
- F: 脱却頻度設定をPhase 1に含めるか、Phase 2以降に先送りするか
- G: lat/lngの保存を廃止するか、仕様書を更新するか

## 論点 4: 脱却モード設計の見直し

### 現状の問題
現在の「興味タグ」方式では、**興味タグに含まれないジャンル＝すべて脱却モード** になる。しかし実際には「特に興味はないが苦でもない」ジャンルが大量に存在し、それらに対しても脱却モードが発動してしまう。

例:
- 興味タグ: カフェ、美術館、公園（3つ）
- 残り19ジャンル: すべて「脱却モード」扱い → ラーメン屋に行くだけで「脱却」になるのは違和感

### 検討すべき設計案

#### 案A: 忌避タグ方式（ユーザー提案）
- 「絶対に行きたくない or 恐ろしく興味がない」ジャンルだけを登録
- **脱却 = 忌避タグに含まれるジャンルへの訪問**
- メリット: 脱却の「怖さ・苦痛」が本物になる
- デメリット: 忌避タグが少なすぎると脱却機会が極端に減る / ゼロ件の場合のハンドリング

#### 案B: 三段階タグ方式
- **好き** / **普通** / **苦手** の3段階でジャンルを分類
- 脱却 = 「苦手」ジャンルへの訪問
- メリット: 精度が高い。XP傾斜も「好き=50, 普通=75, 苦手=100」のように段階的にできる
- デメリット: オンボーディングの設定コストが上がる（22ジャンル × 3段階）

#### 案C: 興味タグ + 忌避タグの併用
- 興味タグ（好き）と忌避タグ（苦手）の両方を設定
- **脱却 = 忌避タグに含まれるジャンルへの訪問**
- **通常 = どちらにも含まれないジャンル + 興味タグのジャンル**
- メリット: A案の「忌避ゼロ問題」を緩和（興味タグはパーソナライズ提案に、忌避タグは脱却判定に使う）
- デメリット: 概念が2つになるのでUI・説明コストが増える

#### 案D: 段階的忌避（推奨検討案）
- 現在の興味タグはパーソナライズ提案の入力として維持
- **脱却モードは「未訪問ジャンル数が多い = コンフォートゾーン外」で自動判定**
- ジャンル熟練度（genre_proficiency）が0のジャンル → 脱却候補
- メリット: ユーザーが明示的にタグ管理しなくてよい（行動ベースで自動変化）
- デメリット: 初期状態で未訪問ジャンルが多いため脱却だらけになる → 初期は熟練度ベースの脱却を無効にし一定量の訪問後に有効化

### 脱却モードの演出強化
- ユーザー提案: 脱却モードのバッジに「危険感を煽るエフェクト」
- 検討: 脱却カード自体の見た目を変える（炎エフェクト、警告色、「DANGER ZONE」表示など）
- 脱却訪問後の演出を通常より派手にする（画面シェイク、特別なアニメーション）

## 論点 5: 興味タグの自動更新・学習サイクル

### 現状の問題
- 興味タグの変更は「設定 → 提案設定 → 興味タグ編集」という深い導線
- ユーザーが能動的に変更しない限り、初期設定のままずっと使い続ける
- 結果:「もう慣れたジャンル」に対してずっと脱却モードが発動 → 不当に高いXPを獲得し続ける

### 検討すべき仕組み

#### A. 訪問後の自動提案（ユーザー提案）
- 脱却モードで訪問した後、「このジャンル、もう怖くない？興味タグに追加する？」と聞く
- ユーザーが「はい」→ 興味タグに自動追加
- ジャンル熟練度が一定レベル（例: Lv.3+）に達したら自動提案

#### B. 定期的な振り返りプロンプト
- 月1回程度、ログイン時に「興味タグを見直しませんか？」と表示
- 訪問頻度の高いジャンルを「追加候補」として提示

#### C. 熟練度ベースの自動卒業（ジャンルマスタリー）
- あるジャンルの熟練度がMaxレベルに達したら、自動的に脱却判定から除外
- 「〇〇ジャンルをマスターしました！もうコンフォートゾーンの一部です」的な演出
- ユーザーの操作なしで自然にタグが進化する

#### D. 脱却XPの減衰
- 同じジャンルの脱却訪問を繰り返すごとにXPボーナスを減少
  - 1回目: 100 XP
  - 2回目: 80 XP
  - 3回目以降: 50 XP（通常と同じに収束）
- メリット: タグ未更新でもXP搾取が自然に止まる
- 注意: 論点4（脱却モードの定義見直し）と合わせて検討する必要あり

## 論点 6: ジャンルマッピングのギャップ（追加観点）

### 概要
seed で定義されている 22 ジャンルのうち、Google Places API の `types` → ジャンル名の変換マッピング（`placeTypeToGenreName`）が **6ジャンルで欠落** している。

### 影響を受けるジャンル

| ジャンル | seedに定義 | APIマッピング | 影響 |
|---------|-----------|-------------|------|
| ラーメン・麺類 | ✅ | ❌ | 興味タグに設定可能だが、提案カードでは「興味マッチ」にならない |
| 海・川・湖 | ✅ | ❌ | 同上 |
| スポーツジム | ✅ | ❌ | 同上（`gym` タイプは存在するがマッピング未定義） |
| カラオケ | ✅ | ❌ | 同上 |
| ゲームセンター | ✅ | ❌ | 同上 |
| 温泉・銭湯 | ✅ | ❌ | 同上 |

### 問題
- これらを興味タグに設定しても**パーソナライズ提案で興味マッチ扱いにならない**
- 常に「脱却モード」扱いになる可能性がある
- ユーザーは「ラーメン好きって設定したのにラーメン屋が提案されない」と感じる

### 対応方針
- Places API の `types` を調査し、対応するタイプを追加する
- 対応する `types` がない場合、そのジャンルをseedから削除するか、Places API (New) の `includedTypes` での細かいフィルタリングを検討する

## 論点 7: 提案の枯渇と多様性の確保（追加観点）

### 概要
同じ地域で毎日使い続けると、訪問済み施設が増えて「新しい施設の提案」がなくなる。

### 想定シナリオ
- 半径10km圏内のvisitableな施設 = 仮に200件
- 1日3件 × 30日 = 月90件消費
- **2〜3ヶ月で提案候補が枯渇する可能性**

### 検討すべきこと
- **枯渇時のフォールバック**: 「近隣に未訪問の施設がありません」と表示? 半径を自動拡大?
- **再訪問の提案**: 一定期間（例: 1ヶ月）経過した施設は再提案対象にする？
- **施設の入れ替わり**: 新規開店・閉店の反映タイミング（Places APIのデータ更新頻度に依存）
- **都市部 vs 地方部**: 地方では初日から施設数が不足する可能性。最低施設数の閾値を定義すべき
- **リロード時の重複**: リロードで同じ施設が再表示されるケースのハンドリング

### 論点1（APIカバレッジ）と合わせて検証する

## 論点 8: XP・レベルバランスの再設計（追加観点）

### 現状の問題
- **レベルキャップが低い**: 実装はLv.10（5,000XP）で打ち止め。通常訪問50XPなら100回（約3ヶ月）でカンスト
- **インフレしやすい**: 脱却モード100XP + 初ジャンル50XP = 150XP/回。脱却が容易なら加速度的にレベルが上がる
- **長期プレイヤーのモチベーション維持手段がない**: カンスト後にXPを貯める意味がなくなる

### 検討すべきこと
- レベルテーブルの拡張（最低Lv.30〜50）
- 称号の実装（仕様書にはあるが未実装）
- ストリークボーナスXPの実装（仕様書にはあるが未実装）
- 初エリアボーナスの有効化（コードは存在するが常にfalse）
- 脱却XPの見直し（論点4・5の設計変更と連動）
- **プレステージ制**の検討: レベルキャップ到達後に「転生」してレベル1からやり直す（別の称号を得る）

## 論点 9: 「提案→訪問」コアループの摩擦ポイント（追加観点）

### 概要
「提案を見る→移動する→到着する→記録する」のフローにおいて、特に「移動中」と「到着・記録」のUXが未考慮。

### 具体的な摩擦ポイント

#### A. 「行く」と「記録する」の分離
- 現在: カードの「行ってきた！」ボタンを押す = 訪問記録完了
- 問題: **実際に行く前に押せてしまう**（位置検証なし）
- 検討:
  - 位置検証を導入するか？（施設から○m以内にいる時のみ記録可能）
  - 自己申告制のままにするか？（MVP的にはこちらが妥当。ゲーミフィケーションの不正行為はPhase 3以降の課題）
  - 「これから行く」ボタン → 到着時に「行ってきた！」の2段階にするか？

#### B. 移動中の体験
- 提案を見て「行こう」と決めたあと、移動中にアプリが何も提供しない
- 検討:
  - Google Maps へのディープリンク（ナビゲーション連携）→ 現時点で実装済み？
  - 移動中の応援メッセージ / Tips 表示（Phase 2の通知機能で対応？）

#### C. 営業時間・定休日の考慮
- 提案された施設が到着時に閉まっている可能性
- Places API の `opening_hours` データを活用して、営業中の施設のみ提案するか
- デメリット: 深夜ユーザー（ナイトウォーカーバッジの対象者）に提案できなくなる可能性(居酒屋などは夜営業なので問題ないだろう)

#### D. 「行ったけど微妙だった」のフィードバック
- 評価（5段階）とメモは実装済みだが、低評価のフィードバックをどう活用するか
- 低評価施設と同系統の施設を今後の提案から除外する？（パーソナライズの精度向上）

## 論点 10: 位置情報の保存方針とプライバシーの矛盾（追加観点）

### 現状の矛盾
- **仕様書** (requirements.md §2.3): 「精密な位置履歴はサーバーに保存しない（Place IDのみ）」
- **実装**: `visit_history` テーブルに `lat`, `lng` カラムが存在し、訪問記録時に保存されている
- **利用箇所**: エリアパイオニアバッジの判定（前回訪問地からの距離計算）、マップ表示

### 検討すべきこと
- **lat/lng は必要か？**
  - エリアパイオニアバッジ: Place IDからGoogle Geocoding APIで座標を取得する方法に変更可能（ただしAPI呼び出しコスト）
  - マップ表示: Place IDからクライアント側で座標を取得可能（ただし表示速度に影響）
- **「精密位置データ」の定義**: Place の座標 ≠ ユーザーの位置。施設の座標は公開情報なので、プライバシー上の問題は小さいとも解釈可能
- **仕様書を更新して実態に合わせるか、実装を変更するか**
- ローンチ前にプライバシーポリシーを明示する必要がある

## 論点 11: 日次制限・リロード回数の妥当性（追加観点）

### 現状
- 1日3件提案 + 最大3回リロード = 最大12件/日の施設を閲覧可能
- すべて訪問しなくても閲覧は可能

### 検討すべきこと
- **3件/日は少なすぎないか？** 休日にまとめて探索したいユーザーの需要
- **リロード3回は多すぎないか？** リロードで「いい施設が出るまでガチャを引く」行動を誘発しないか
- 「提案は3件だが訪問は何件でも記録可能」にする案（提案外の施設もPlace IDで手動記録）
- Premium で日次上限を緩和する設計（Phase 3フリーミアムモデルとの整合性）

## 論点 12: Premium プランの価値設計（追加観点）

### 現状の課題
仕様書のPremiumプラン（¥580/月）の提供価値:
- AI提案無制限（Phase 2のGemini連携が前提）
- 詳細な行動分析レポート
- プレミアムバッジ
- 高度なフィルター

**問題: Phase 1時点でFreeプランの体験が十分すぎるため、Premiumの差別化が弱い。**

### 検討すべきこと（ローンチ後でも可だが設計には影響）
- Freeで十分に価値を感じさせつつ「もっと欲しい」を作れるか
- リロード回数の上限（Free: 3回/日 → Premium: 無制限）が課金のトリガーになるか
