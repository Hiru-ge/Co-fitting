## 論点 1: Google Maps API の施設カバレッジ

### 概要
実際に存在する施設のうち Google Places API で呼び出せるものがどれだけあるか。「Roamble で提案できる施設」＝「Places API に登録されている施設」なので、カバレッジが低ければプロダクト価値そのものに直結する。

### 調査すべきこと
- **都市部**（東京・大阪等）と**地方部**での施設数の差
- **ジャンル別のカバレッジ差**: 飲食（カフェ・レストラン）は豊富だが、ニッチなジャンル（ゲームセンター・温泉など）はどうか
- **APIの検索タイプ制限**: 現在 `visitableTypes`（28タイプ）でフィルタしているが、このリストの妥当性
- **チェーン店の割合**: 提案がコンビニ・チェーン飲食店だらけにならないか
- **営業時間データの精度**: Places APIで営業中かどうかの判定がどこまで正確か
- **実態調査**: 実際にユーザーの近所の施設をAPIで検索し、近くに存在するのに提案されない施設がどれくらいあるのか、ユーザーの体験に即した調査を行う

### 検証方法（案）
- 実際に数地点（都心・郊外・地方）で API を叩き、返却される施設数・種類を比較する
- 結果を「ジャンル × エリア」のマトリクスで整理する
- 開発者の現在地でAPIを叩いて周辺3kmの全施設を一旦リストアップし、実態との乖離を確認する
- ユーザー(開発者含む)からのフィードバックを集め、実際に「近所のあの店が提案されない」という声がどれくらいあるかを定量化する

### ✅ 議論結果 (2026-02-27)

**根本原因**: 旧 Places API の `NearbySearch` は1回あたり最大20件・人気度順のため、小規模な地元施設がヒットしない。

**決定方針: New Places API（v1）に移行する**
- `rankPreference: DISTANCE`（距離順）に変更 → 近くの地元施設を優先して取得
- `includedTypes` で visitableTypes を明示指定（ノイズ除去）
- `fieldMask` に rating + photos を含める（Enterprise SKU = $35/1,000、旧API $40/1,000 より安い）
- Go SDK は旧API専用のため `net/http` で REST API を直接呼び出すよう変更
- 詳細: Issue #196 / docs/todo.md 参照

## 論点 2: 仕様書と現コードの認識齟齬

### 概要
要件定義書（requirements.md）と実装コードの間にある乖離を特定・解消する。

## 論点 4: 脱却モード設計の見直し

### 現状の問題
現在の「興味タグ」方式では、**興味タグに含まれないジャンル＝すべて脱却モード** になる。しかし実際には「特に興味はないが苦でもない」ジャンルが大量に存在し、それらに対しても脱却モードが発動してしまう。

例:
- 興味タグ: カフェ、美術館、公園（3つ）
- 残り19ジャンル: すべて「脱却モード」扱い → ラーメン屋に行くだけで「脱却」になるのは違和感

### 検討すべき設計案

#### 案A: 忌避タグ方式（ユーザー提案）
- 「絶対に行きたくない or 恐ろしく興味がない」ジャンルだけを登録
- **脱却 = 忌避タグに含まれるジャンルへの訪問**
- メリット: 脱却の「怖さ・苦痛」が本物になる
- デメリット: 忌避タグが少なすぎると脱却機会が極端に減る / ゼロ件の場合のハンドリング

#### 案B: 三段階タグ方式
- **好き** / **普通** / **苦手** の3段階でジャンルを分類
- 脱却 = 「苦手」ジャンルへの訪問
- メリット: 精度が高い。XP傾斜も「好き=50, 普通=75, 苦手=100」のように段階的にできる
- デメリット: オンボーディングの設定コストが上がる（22ジャンル × 3段階）

#### 案C: 興味タグ + 忌避タグの併用
- 興味タグ（好き）と忌避タグ（苦手）の両方を設定
- **脱却 = 忌避タグに含まれるジャンルへの訪問**
- **通常 = どちらにも含まれないジャンル + 興味タグのジャンル**
- メリット: A案の「忌避ゼロ問題」を緩和（興味タグはパーソナライズ提案に、忌避タグは脱却判定に使う）
- デメリット: 概念が2つになるのでUI・説明コストが増える

#### 案D: 段階的忌避（推奨検討案）
- 現在の興味タグはパーソナライズ提案の入力として維持
- **脱却モードは「未訪問ジャンル数が多い = コンフォートゾーン外」で自動判定**
- ジャンル熟練度（genre_proficiency）が0のジャンル → 脱却候補
- メリット: ユーザーが明示的にタグ管理しなくてよい（行動ベースで自動変化）
- デメリット: 初期状態で未訪問ジャンルが多いため脱却だらけになる → 初期は熟練度ベースの脱却を無効にし一定量の訪問後に有効化

### 脱却モードの演出強化
- ユーザー提案: 脱却モードのバッジに「危険感を煽るエフェクト」
- 検討: 脱却カード自体の見た目を変える（炎エフェクト、警告色、「DANGER ZONE」表示など）
- 脱却訪問後の演出を通常より派手にする（画面シェイク、特別なアニメーション）

### ✅ 議論結果 (2026-02-27)

**決定方針: 案D採用**

- **脱却 = ジャンル熟練度Lv.1（100XP未満）のジャンルへの訪問**
- **通常 = 熟練度Lv.2以上になったジャンルへの訪問**（2回目以降の訪問で熟練度が上がる）
- 興味タグはパーソナライズ提案専用として維持（忌避タグは実装しない）
- 将来改善（Phase 2以降）: 「最近訪問していないジャンル」なども脱却候補に追加検討

詳細: Issue #198 / docs/todo.md 参照

## 論点 5: 興味タグの自動更新・学習サイクル

### 現状の問題
- 興味タグの変更は「設定 → 提案設定 → 興味タグ編集」という深い導線
- ユーザーが能動的に変更しない限り、初期設定のままずっと使い続ける
- 結果:「もう慣れたジャンル」に対してずっと脱却モードが発動 → 不当に高いXPを獲得し続ける

### 検討すべき仕組み

#### A. 訪問後の自動提案（ユーザー提案）
- 脱却モードで訪問した後、「このジャンル、もう怖くない？興味タグに追加する？」と聞く
- ユーザーが「はい」→ 興味タグに自動追加
- ジャンル熟練度が一定レベル（例: Lv.3+）に達したら自動提案

#### B. 定期的な振り返りプロンプト
- 月1回程度、ログイン時に「興味タグを見直しませんか？」と表示
- 訪問頻度の高いジャンルを「追加候補」として提示

#### C. 熟練度ベースの自動卒業（ジャンルマスタリー）
- あるジャンルの熟練度がMaxレベルに達したら、自動的に脱却判定から除外
- 「〇〇ジャンルをマスターしました！もうコンフォートゾーンの一部です」的な演出
- ユーザーの操作なしで自然にタグが進化する

#### D. 脱却XPの減衰
- 同じジャンルの脱却訪問を繰り返すごとにXPボーナスを減少
  - 1回目: 100 XP
  - 2回目: 80 XP
  - 3回目以降: 50 XP（通常と同じに収束）
- メリット: タグ未更新でもXP搾取が自然に止まる
- 注意: 論点4（脱却モードの定義見直し）と合わせて検討する必要あり

### ✅ 議論結果 (2026-02-27)

**論点4で案Dを採用したことで、脱却判定が興味タグから切り離された。**

- 脱却判定が「熟練度Lv.1かどうか」になったため、興味タグの自動更新は脱却制御に直結しなくなった
- 興味タグはパーソナライズ提案の精度向上のためにのみ使われるため、自動更新の緊急度は低い
- **Phase 2以降に先送り**（Phase 1では手動更新のみ）
- 将来検討: 熟練度Lv.3以上に達したジャンルを「興味タグ候補」として提案するUIなど

## 論点 6: ジャンルマッピングのギャップ（追加観点）

### 概要
seed で定義されている 22 ジャンルのうち、Google Places API の `types` → ジャンル名の変換マッピング（`placeTypeToGenreName`）が **6ジャンルで欠落** している。

### 影響を受けるジャンル

| ジャンル | seedに定義 | APIマッピング | 影響 |
|---------|-----------|-------------|------|
| ラーメン・麺類 | ✅ | ❌ | 興味タグに設定可能だが、提案カードでは「興味マッチ」にならない |
| 海・川・湖 | ✅ | ❌ | 同上 |
| スポーツジム | ✅ | ❌ | 同上（`gym` タイプは存在するがマッピング未定義） |
| カラオケ | ✅ | ❌ | 同上 |
| ゲームセンター | ✅ | ❌ | 同上 |
| 温泉・銭湯 | ✅ | ❌ | 同上 |

### 問題
- これらを興味タグに設定しても**パーソナライズ提案で興味マッチ扱いにならない**
- 常に「脱却モード」扱いになる可能性がある
- ユーザーは「ラーメン好きって設定したのにラーメン屋が提案されない」と感じる

### 対応方針
- Places API の `types` を調査し、対応するタイプを追加する
- 対応する `types` がない場合、そのジャンルをseedから削除するか、Places API (New) の `includedTypes` での細かいフィルタリングを検討する

### ✅ 議論結果 (2026-02-27)

**New Places API（v1）では全6ジャンルのマッピングが可能**（削除不要）:

| ジャンル | New API type |
|---|---|
| ラーメン・麺類 | `ramen_restaurant` |
| カラオケ | `karaoke` |
| ゲームセンター | `amusement_center`, `video_arcade` |
| 温泉・銭湯 | `public_bath`, `sauna` |
| **スポーツジム** | `gym`（visitableTypes 登録済みだが placeTypeToGenreName が欠落 → **即時バグ修正**） |
| 海・川・湖 | `beach`, `lake`, `river` |

**緊急度が高いアクション**: スポーツジムのマッピングバグは今すぐ修正可能 → Issue #195
**中期アクション**: New API 移行時に残5ジャンルのマッピングを追加 → Issue #196

## 論点 10: 位置情報の保存方針とプライバシーの矛盾（追加観点）

### 現状の矛盾
- **仕様書** (requirements.md §2.3): 「精密な位置履歴はサーバーに保存しない（Place IDのみ）」
- **実装**: `visit_history` テーブルに `lat`, `lng` カラムが存在し、訪問記録時に保存されている
- **利用箇所**: エリアパイオニアバッジの判定（前回訪問地からの距離計算）、マップ表示

### 検討すべきこと
- **lat/lng は必要か？**
  - エリアパイオニアバッジ: Place IDからGoogle Geocoding APIで座標を取得する方法に変更可能（ただしAPI呼び出しコスト）
  - マップ表示: Place IDからクライアント側で座標を取得可能（ただし表示速度に影響）
- **「精密位置データ」の定義**: Place の座標 ≠ ユーザーの位置。施設の座標は公開情報なので、プライバシー上の問題は小さいとも解釈可能
- **仕様書を更新して実態に合わせるか、実装を変更するか**
- ローンチ前にプライバシーポリシーを明示する必要がある

### ✅ 議論結果 (2026-02-27)

**論点2の議論結果と統合済み。**

- lat/lng は「施設の公開座標」であり、ユーザーの精密な移動履歴ではないため、プライバシーリスクは小さいと判断
- **実装は変えない**（lat/lngをvisit_historyに保存するロジックを維持）
- **仕様書（requirements.md）を実態に合わせて更新**（リスク表の「精密位置データは端末のみに保持」を修正済み）
