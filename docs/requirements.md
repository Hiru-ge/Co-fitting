# Questerra — 要件定義書

コンフォートゾーン脱却支援Webアプリケーション

---

## 1. プロダクト概要

### 1.1 ビジョン

**「日常の行動圏を冒険のフィールドに変える」**

Questerraは、ユーザーが普段の行動パターンから一歩踏み出し、新しい場所・体験に出会うことを支援するアプリケーションである。AIによるパーソナライズ提案とゲーミフィケーションを組み合わせ、「行動変容」を楽しく継続可能にする。

### 1.2 コア価値

| 価値 | 説明 |
|------|------|
| **発見** | 普段行かない場所・ジャンルとの出会いを創出する |
| **成長** | 訪問履歴の可視化とゲーミフィケーションで成長実感を提供する |
| **パーソナライズ** | AIが行動パターンと気分を分析し、最適な「一歩先」を提案する |
| **継続性** | ストリーク・XP・バッジで習慣化を支援する |

### 1.3 ターゲットユーザー

- **メインターゲット**: 幅広い年齢層（学生〜社会人）で、マンネリを感じている人
- **ペルソナ例**:
  - 毎日同じカフェに通う会社員が、新しい店を開拓したい
  - 休日の過ごし方がワンパターンな大学生が、新しい趣味・場所を探したい
  - 引っ越し先の街をもっと知りたい新生活者

## 2. 競合分析

### 2.1 競合マップ

| サービス | カテゴリ | 主な機能 | Questerraとの差別化 |
|---------|---------|---------|-------------------|
| **Randonautica** | ランダム探索 | 量子乱数で座標を生成し冒険を促す | ランダム性のみで「パーソナライズ」がない。私有地や水域に案内されることも。Questerraは実在の施設をAIで提案 |
| **Habitica** | ゲーミフィケーション習慣 | RPG要素でタスク・習慣をゲーム化 | 汎用的な習慣管理であり「場所の探索」には非対応。Questerraは場所特化のゲーミフィケーション |
| **Swarm (Foursquare)** | チェックイン・記録 | 訪問記録・リスト作成・チェックインログ | 記録ツールであり「提案」機能がない。Questerraは提案+記録+ゲーミフィケーションの統合体験 |
| **SPARKFUL** | マイクロチャレンジ | 日常の小さな挑戦を提案 | 場所に特化していない汎用チャレンジ。Questerraは位置情報ベースで具体的な場所を提案 |
| **AI旅行プランナー (Mindtrip等)** | AI旅行計画 | AI で旅行先を提案・計画 | 旅行向けであり日常使いではない。Questerraは「日常圏内の冒険」に特化 |

### 2.2 Questerraの市場ポジショニング

```
                提案・発見
                    ▲
                    │
       Randonautica │  ★ Questerra
      (ランダム)    │  (AI + ゲーミフィケーション)
                    │
  日常 ◄────────────┼────────────► 旅行
                    │
  Swarm             │  Mindtrip
  (記録のみ)        │  (旅行計画)
                    │
                    ▼
                記録・整理
```

### 2.3 差別化戦略

1. **「日常圏の冒険」という独自ポジション**: 旅行でもなく、完全ランダムでもない。ユーザーの生活圏内で「ちょうどいい一歩先」を提案する
2. **AI × ゲーミフィケーションの統合**: 行動パターン分析による提案精度と、Duolingoライクな継続メカニクスの組み合わせ
3. **気分ベースの提案**: 「リラックスしたい」「体を動かしたい」等の気分選択でコンテキストを加味

## 3. 機能要件

### 3.1 フェーズ分け

| フェーズ | 名称 | 概要 |
|---------|------|------|
| **Phase 1 (MVP)** | コア体験 | 提案・訪問記録・基本ゲーミフィケーション |
| **Phase 2** | AI強化・通知 | AI提案の高度化・プッシュ通知・リマインダー |
| **Phase 3** | ソーシャル・プレミアム | フレンド機能・ランキング・フリーミアムモデル |
| **Phase 4** | iOSネイティブ | iOSアプリ化（技術選定は後日判断） |

---

### 3.2 Phase 1 — MVP機能

#### A. ユーザー認証・管理

| 項目 | 詳細 |
|------|------|
| 認証方式 | メール/パスワード + Google OAuth |
| プロフィール | 表示名、アバター画像、自己紹介（任意） |
| 初回セットアップ | 興味タグの選択（オンボーディングフロー） |

#### B. 提案・検索機能

| 項目 | 詳細 |
|------|------|
| 位置情報取得 | ブラウザのGeolocation APIで現在地を取得 |
| 施設検索 | Google Maps Places APIで周辺施設を検索 |
| 通常モード | ユーザーの興味タグに合致する施設を提案 |
| 脱却モード | 興味タグ「外」のジャンルをあえて提案。脱却度に応じた段階的な提案（少し外す〜大きく外す） |
| 気分選択 | 「リラックス」「体を動かしたい」「刺激がほしい」「学びたい」等のプリセットから選択し、提案に反映 |
| 除外ロジック | 訪問済みの場所は候補から除外 or 優先度低下 |
| 提案表示 | カード形式で施設情報（名前・写真・ジャンル・距離・評価）を表示。スワイプで次の提案に切替 |

#### C. 記録・可視化機能

| 項目 | 詳細 |
|------|------|
| 訪問記録 | 「行った！」ボタンで訪問を記録。施設名・ジャンル・日時を保存 |
| 感想メモ | 訪問時に一言メモ（任意）と5段階評価を入力可能 |
| マップ表示 | Google Maps上に訪問済みピンを表示。エリアごとの制覇率を可視化 |
| タイムライン | 訪問履歴を時系列で閲覧可能 |

#### D. ゲーミフィケーション

| 要素 | 詳細 |
|------|------|
| **XP (経験値)** | 訪問ごとにXPを付与。脱却モードの訪問はボーナスXP |
| **ジャンル熟練度** | カフェ・美術館・公園等、ジャンルごとにXP・レベルを管理 |
| **ユーザーレベル** | 総XPに応じたレベルアップ。レベルに応じた称号を付与 |
| **ストリーク** | 週に1回以上訪問すると継続。ストリークの長さに応じてボーナスXP |
| **バッジ** | 特定条件で付与（例: 「初脱却」「3ジャンル制覇」「10連続ストリーク」「全く新しいエリア訪問」） |
| **進捗ダッシュボード** | レベル・XP・ストリーク・バッジを一覧表示するホーム画面 |

#### E. ユーザー設定

| 項目 | 詳細 |
|------|------|
| 興味タグ管理 | 興味ジャンルの追加・削除 |
| 検索半径設定 | 提案の検索範囲を設定（例: 1km / 3km / 5km / 10km） |
| 脱却頻度設定 | 脱却モードの発動頻度を調整（低・中・高） |

### 3.3 Phase 2 — AI強化・通知

#### A. AI提案機能（Gemini API連携）

| 項目 | 詳細 |
|------|------|
| 行動パターン分析 | 訪問履歴からジャンル偏り・時間帯・頻度を分析 |
| パーソナライズ提案 | 「あなたはカフェが多いので、今日は美術館はいかが？」等の文脈ある提案 |
| 気分 × 履歴の掛け合わせ | 選択した気分と過去の行動パターンを組み合わせた提案生成 |
| 提案理由の説明 | なぜこの場所を提案したかをAIが自然言語で説明 |

#### B. 通知・リマインダー

| 項目 | 詳細 |
|------|------|
| ストリークリマインダー | ストリーク切れが近い場合に通知 |
| 週次サマリー | 週の訪問数・獲得XP・新ジャンル挑戦をメールで通知 |
| おすすめ通知 | 天気や曜日に合わせた提案プッシュ通知（Web Push API） |

## 4. 非機能要件

### 4.1 パフォーマンス

| 項目 | 目標値 |
|------|--------|
| ページ読み込み | 初回3秒以内、以降1秒以内（LCP） |
| API応答時間 | 95パーセンタイルで500ms以内 |
| 提案生成時間 | AI提案含め3秒以内 |

### 4.2 可用性

| 項目 | 目標値 |
|------|--------|
| 稼働率 | 99.5%以上 |
| 計画停止 | 月1回以内、深夜帯に実施 |

### 4.3 セキュリティ

| 項目 | 詳細 |
|------|------|
| 通信 | 全通信をHTTPS化 |
| 認証トークン | JWT（アクセストークン + リフレッシュトークン） |
| パスワード | bcryptでハッシュ化 |
| 位置情報 | 明示的な同意のもと取得。精密な位置履歴はサーバーに保存しない（訪問場所のPlace IDのみ保存） |
| 個人情報保護 | GDPR / 個人情報保護法に準拠。データエクスポート・削除機能を実装 |

### 4.4 スケーラビリティ

| 項目 | 詳細 |
|------|------|
| 初期想定 | DAU 1,000人 |
| スケール目標 | DAU 100,000人まで対応可能な設計 |
| アーキテクチャ | ステートレスなAPIサーバー + マネージドDB でスケールアウト可能に |

## 7. API設計（主要エンドポイント）

> **バージョニング方針**: MVP段階ではAPIのバージョニング（`/v1/`プレフィックス等）を行わない。自分のみが利用するAPIであり、バージョン管理のオーバーヘッドに見合わない。iOS化等でクライアントが複数になり破壊的変更が必要になった場合に、URLパスバージョニングを導入する。

### 7.1 認証

| Method | Path | 説明 |
|--------|------|------|
| POST | `/api/auth/signup` | メール+パスワードで新規登録 |
| POST | `/api/auth/login` | メール+パスワードでログイン |
| POST | `/api/auth/oauth/google` | Google OAuthコールバック |
| POST | `/api/auth/refresh` | トークンリフレッシュ |
| POST | `/api/auth/logout` | ログアウト |

### 7.2 ユーザー

| Method | Path | 説明 |
|--------|------|------|
| GET | `/api/users/me` | 自分のプロフィール取得 |
| PATCH | `/api/users/me` | プロフィール更新 |
| GET | `/api/users/me/stats` | レベル・XP・ストリーク等の統計 |
| DELETE | `/api/users/me` | アカウント削除 |

### 7.3 興味タグ

| Method | Path | 説明 |
|--------|------|------|
| GET | `/api/genres` | ジャンルタグ一覧取得 |
| GET | `/api/users/me/interests` | 自分の興味タグ取得 |
| PUT | `/api/users/me/interests` | 興味タグを一括更新 |

### 7.4 提案

| Method | Path | 説明 |
|--------|------|------|
| POST | `/api/suggestions` | 場所の提案を取得（位置情報・気分を送信） |
| POST | `/api/suggestions/:id/dismiss` | 提案を却下（フィードバック） |

### 7.5 訪問記録

| Method | Path | 説明 |
|--------|------|------|
| GET | `/api/visits` | 訪問履歴一覧（ページネーション） |
| POST | `/api/visits` | 訪問を記録 |
| GET | `/api/visits/:id` | 訪問詳細取得 |
| PATCH | `/api/visits/:id` | 訪問記録の編集（メモ・評価） |
| GET | `/api/visits/map` | マップ表示用の訪問データ |

### 7.6 ゲーミフィケーション

| Method | Path | 説明 |
|--------|------|------|
| GET | `/api/users/me/badges` | 獲得済みバッジ一覧 |
| GET | `/api/badges` | バッジ一覧（未獲得含む） |
| GET | `/api/users/me/proficiency` | ジャンル別熟練度一覧 |

## 9. XP・レベル設計（Duolingoモデル参考）

### 9.1 XP付与ルール

| アクション | XP |
|-----------|-----|
| 通常訪問（興味ジャンル内） | 50 XP |
| 脱却訪問（興味ジャンル外） | 100 XP |
| 初めてのジャンル訪問 | +50 XP ボーナス |
| 初めてのエリア訪問 | +30 XP ボーナス |
| ストリークボーナス（週次） | 連続週数 × 10 XP |
| 感想メモ記入 | +10 XP |

### 9.2 レベルテーブル（案）

| レベル | 必要累計XP | 称号 |
|--------|-----------|------|
| 1 | 0 | 旅立ちの冒険者 |
| 5 | 1,000 | 街の探検家 |
| 10 | 3,000 | 好奇心の開拓者 |
| 20 | 10,000 | 未知の征服者 |
| 30 | 25,000 | 伝説の冒険家 |
| 50 | 100,000 | Questerra マスター |

※ レベル間の必要XPは段階的に増加させる（指数カーブ）

### 9.3 バッジ例

## 10. iOS化に向けた考慮事項

### 10.1 Web版での準備

| 項目 | 対応 |
|------|------|
| API設計 | RESTful APIをWeb/iOS両対応で設計 |
| 認証 | OAuth + JWT方式でプラットフォーム非依存 |
| ビジネスロジック | バックエンド集約でフロントは表示に徹する |
| レスポンシブ | モバイルファーストでUI設計（iOS化時の操作感を事前検証） |

### 10.2 技術選定の判断基準（後日決定）

| 選択肢 | メリット | デメリット |
|--------|---------|----------|
| **React Native** | Reactの知見を活用可能。Web版コードの一部再利用 | ネイティブ機能のブリッジが必要な場合がある |
| **Swift (ネイティブ)** | 最高のパフォーマンス・UX。Apple APIとの親和性 | Web版と別コードベース。開発コスト高 |
| **Flutter** | クロスプラットフォーム対応。高パフォーマンス | Dart習得が必要。Reactの知見が活かせない |
| **PWA** | 追加開発不要。Web版をそのまま配布可能 | プッシュ通知やバックグラウンド処理に制限あり |

### 10.3 iOS固有の要件（将来）

- Apple Sign-In対応（Google OAuthを提供する場合、App Storeガイドライン4.8により同等のプライバシー保護を持つ代替ログインが必要。Apple Sign-Inが最も簡便な対応策）
- App Tracking Transparency対応
- バックグラウンド位置情報の適切なハンドリング
- App Store審査ガイドラインへの準拠
- アプリ内課金（In-App Purchase）でのプレミアムプラン提供

## 11. リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| Google Maps API費用増大 | コスト増 | キャッシュ戦略の導入。リクエスト数の上限管理 |
| Gemini API の応答遅延 | UX低下 | タイムアウト設定 + フォールバック（ルールベース提案） |
| 位置情報の精度不足 | 提案品質低下 | ユーザーによる手動位置調整を許可 |
| ユーザーの継続率低下 | DAU減 | ストリーク・通知・ゲーミフィケーション要素で離脱防止 |
| 個人情報漏洩 | 信用失墜 | 最小限のデータ保持。精密位置データは端末のみに保持 |
