## 0. MakersGuild知見の適用方針

本要件定義書はMakersGuildの以下の原則に基づいて設計している。

| 原則 | 適用 |
|------|------|
| **1.1 「売れるもの」を作る** | 自身の「新しい場所に行くのが怖い」課題をドッグフーディングで検証 |
| **1.2 課題感 × 予算 × タイミング** | アイデア3条件スコアリングを実施（後述） |
| **1.4 ニッチを制す** | 初期ターゲットを「都市部の20-30代・行動圏が固定化した社会人」に絞り込み |

| **フィードバックを得るためのミニマム** | Phase 0（最小公開）を新設。作りながら検証する |

## 1. プロダクト概要

### 1.1 ビジョン

**「日常の行動圏を冒険のフィールドに変える」**

Roambleは、ユーザーが普段の行動パターンから一歩踏み出し、新しい場所・体験に出会うことを支援するアプリケーションである。AIによるパーソナライズ提案とゲーミフィケーションを組み合わせ、「行動変容」を楽しく継続可能にする。

### 1.2 解決する課題

> 「新しい場所に行きたいけど、知らない店に一人で入るのが怖い。結局いつもの場所に落ち着いてしまう」

- **課題の頻度**: 毎週末（週1〜2回）発生する日常的な課題
- **課題の深さ**: 「知らない場所への心理的ハードル」は行動を実際に制限しており、慢性的な停滞感として蓄積する。単なる「めんどくさい」より深い
- **課題の二層構造**: (1) 心理的障壁 — 知らない場所・初めての店に入るのが怖い (2) 探索コスト — 自分で探すと情報過多で疲れる。この2つが重なり「いつもの場所」に収束する
- **既存の解決策の不足**: Google Mapsは「行きたい場所が決まっている人」向け。「何が良いかわからない、しかも入るのが怖い」人には機能しない

### 1.3 アイデア3条件スコアリング（MakersGuild 1.2準拠）

| 条件 | 評価 | 根拠 |
|------|------|------|
| **課題感** | ★★★☆☆ | 「死ぬほど困る」ではないが、行動を実際に制限している慢性的な課題。「怖い」は「めんどくさい」より課題感が深い。ゲーミフィケーションで成功体験を積み重ね、心理的ハードルを下げる |
| **予算** | ★★☆☆☆ | B2C。無料ユーザーが大半になる。フリーミアムで月500〜1,000円程度の課金を狙う |
| **タイミング** | ★★★★☆ | コロナ後の外出回帰トレンド。AI提案の一般化。ゲーミフィケーションアプリの成熟 |

**総合判断**: 課題感と予算が弱めだが、ゲーミフィケーションによる継続性と「楽しさ」で補完する戦略。学習目的も兼ねているため、Go。

### 1.4 コア価値

| 価値 | 説明 |
|------|------|
| **発見** | 普段行かない場所・ジャンルとの出会いを創出する |
| **成長実感** | XP・レベル・バッジで「冒険の軌跡」を可視化する |
| **適度な背中押し** | 「自分で探す→自分で決める→自分で入る」の心理的負荷を、提案によって軽減する。完全ランダムではなく「ちょうどいい一歩先」を提案 |

### 1.5 アハモーメント（MakersGuild 3.4準拠）

> **「提案された場所に"怖いけど行ってみよう"と思えて、実際に行ったら良い場所だった。自分のコンフォートゾーンが広がった実感を得た。」**

この体験を**初回利用時に**到達させることが最重要。「怖かったけど行けた」という成功体験が次の行動への自信になる。Phase 0では提案精度に注力し、Phase 1でゲーミフィケーション（XP獲得演出・レベルアップ）を加えて「もう1回」を誘発する。

## 2. ターゲット設定（MakersGuild 1.4準拠）

### 2.1 初期ニッチ: ドッグフーディング起点

**将来的には幅広い層を狙うが、MVP段階では自分自身の課題感を起点にターゲットを絞る。**

#### プロトタイプペルソナ（= 開発者自身）

| 項目 | 詳細 |
|------|------|
| 属性 | 都市部在住、20-30代、学生・社会人 |
| 課題 | 新しい場所に行きたいが、知らない店に一人で入るのが怖い。結局いつもの場所に落ち着く |
| 行動 | 気になる店の前を素通りしてチェーン店に入る。Google Mapsで探すと情報過多で疲れる。散歩ルートも固定化 |
| 欲求 | 「ここ行ってみたら？」と背中を押してくれる存在がほしい。一歩踏み出した記録を残したい |
| 支払い意思 | 月500〜1,000円なら出せる |

### 2.2 ニッチ市場評価（MakersGuild 1.4準拠）

| 評価項目 | スコア | 根拠 |
|---------|--------|------|
| 市場の明確さ | 4/5 | 「コンフォートゾーン脱却 × 場所提案」は明確 |
| 競合の少なさ | 4/5 | 「日常圏 × AI提案 × ゲーミフィケーション」の組み合わせは未開拓 |
| 課題の深さ | 3/5 | 「怖い」は行動を実際に制限しており、単なるマンネリより深い。ただし生死に関わるレベルではない |
| 到達可能性 | 3/5 | X/Reddit/Product Huntで到達可能だが、ニッチコミュニティが分散 |
| 拡張性 | 4/5 | 旅行・グルメ・フィットネスなど隣接市場への拡張余地あり |

**合計: 18/25** → 有望ライン（15点以上）をクリア

### 2.3 ニッチ → 拡大の戦略

```
Phase 0: 自分自身で検証
    ↓
Phase 1~2:   都市部の20-30代・行動圏が固定化した社会人（Product Hunt / X でリーチ）
    ↓
Phase 3:   新生活者（引っ越し直後）、旅行好き、街歩き趣味層に拡大
    ↓
Phase 4:   幅広い年齢層（iOS展開）
```

## 3. 競合分析（深掘り版）

### 3.1 競合マップ

| サービス | カテゴリ | 主な機能 | ユーザー規模 | Roambleとの差別化 |
|---------|---------|---------|-------------|-------------------|
| **Randonautica** | ランダム探索 | 量子乱数で座標を生成し冒険を促す | App Store評価4.0, 有料化で減少傾向 | ランダム性のみで「パーソナライズ」がない。私有地や水域に案内される問題。Roambleは実在施設をAI提案 |
| **世界の霧 (Fog of World)** | マップ探索ゲーム | 訪問済みの場所の「霧」が晴れる可視化 | App Store評価4.5, 1,700+レビュー, 137カ国でApp of the Day | 「記録・可視化」に特化し「提案」がない。Roambleは提案→行動→可視化の全ループをカバー |
| **Habitica** | ゲーミフィケーション習慣 | RPG要素でタスク・習慣をゲーム化 | 400万+ユーザー | 汎用的な習慣管理で「場所」に非対応。Roambleは場所特化のゲーミフィケーション |
| **Swarm (Foursquare)** | チェックイン・記録 | 訪問記録・リスト・チェックイン | 衰退傾向（2024年サービス縮小） | 記録ツールで「提案」がない。ゲーミフィケーション要素も縮小中 |
| **DareCircle / IDareTo** | コンフォートゾーンチャレンジ | 社会的恐怖を克服するチャレンジを提案 | ニッチ規模 | 心理的チャレンジ（人前で歌うなど）に特化。場所探索ではない |
| **ExplorR** | 位置ゲーム | ポケモンGOライクな都市探索ゲーム | 開発段階 | 観光向けゲーム。日常圏のマンネリ解消ではない |

### 3.2 市場ポジショニング

```
                 AI提案・発見
                     ▲
                     │
        Randonautica │  ★ Roamble
       (完全ランダム) │  (パーソナライズ + ゲーミフィケーション)
                     │
  日常 ◄─────────────┼─────────────► 旅行・観光
                     │
  世界の霧 / Swarm   │  Mindtrip / ExplorR
  (記録・可視化のみ)  │  (旅行計画)
                     │
                     ▼
                 記録・整理
```

### 3.3 差別化戦略

1. **「日常圏のコンフォートゾーン脱却」という独自ポジション**: 旅行でもなく、完全ランダムでもない。「知らない場所に行くのが怖い」を克服する背中押し
2. **提案→行動→報酬の完結ループ**: 世界の霧は「記録のみ」、Randonauticaは「座標生成のみ」。Roambleは提案から報酬まで一気通貫。「怖かったけど行けた」を成功体験として蓄積
3. **ゲーミフィケーション × 行動変容**: Habiticaの継続メカニクスを場所探索に特化適用。「怖い → 行けた → XP獲得」のループで心理的ハードルを段階的に下げる

## 4. 機能要件

### 4.1 フェーズ分け（MakersGuild MVP思考準拠）

| フェーズ | 名称 | 方針 | 主な機能 |
|---------|------|------|---------|
| **Phase 0** | 最小公開 | Build in Publicに留めて本番環境公開しない。少しでもフィードバック獲得しながら「セキュリティ的に出しても問題ないもの」を作るまで耐える | 提案 + 訪問記録 + 簡易認証（最低限） |
| **Phase 1** | コア完成 | Phase 0のフィードバックを反映して磨き上げ。本番公開 | Google OAuth + XP/レベル/ゲーミフィケーション + 興味タグ/パーソナライズ + バッジ + ストリーク + マップ可視化 |
| **Phase 2** | AI強化・通知 | ユーザーが定着したらAIを強化 | Gemini AI提案 + プッシュ通知 |
| **Phase 3** | ソーシャル・プレミアム | 成長フェーズ | フレンド + ランキング + フリーミアムモデル |
| **Phase 4** | iOSネイティブ | プラットフォーム拡大 | iOSアプリ化（技術選定は後日判断） |

### 4.2 Phase 0 — 最小公開（Minimum Viable Launch）

> **MakersGuildの原則**: 「少し恥ずかしい」程度（完成度70%前後）で公開するのがちょうど良いタイミング。機能は1〜2本線で十分。フィードバックが一点集中して鋭くなる。

#### コア機能（これだけ作る）

**A. 提案機能**

| 項目 | 詳細 |
|------|------|
| 位置情報取得 | ブラウザのGeolocation APIで現在地を取得 |
| 施設検索 | Google Maps Places APIで周辺施設を検索 |
| 提案ロジック | 現在地周辺の施設をカード表示。パーソナライズなし（ジャンルフィルター程度） |
| 提案表示 | カード形式（施設名・写真・ジャンル・距離）。スワイプで次の提案 |
| 除外ロジック | 訪問済みの場所は除外 |

**B. 訪問記録（最低限）**

| 項目 | 詳細 |
|------|------|
| 記録 | 「行った！」ボタンで訪問を記録 |
| 履歴 | 訪問履歴の一覧表示（リスト形式） |

**C. 認証（最低限）**

| 項目 | 詳細 |
|------|------|
| 認証方式 | メール+パスワード認証（JWT） |
| パスワード保存 | bcryptでハッシュ化 |

#### Phase 0で「やらない」もの

| 機能 | 理由 | いつやるか |
|------|------|-----------|
| Google OAuth | 簡易認証で十分。Phase 0は自分のみの利用 | Phase 1 |
| XP / レベル / プログレスバー | コアループ「提案→行動」の検証が先。ゲーミフィケーションは上に乗せる層 | Phase 1 |
| 興味タグ選択 / パーソナライズ提案 | タグなしでも「近くの面白い場所」の提案価値は検証可能 | Phase 1 |
| オンボーディング画面 | 興味タグ選択がなくなるため不要 | Phase 1 |
| 脱却モード（興味外ジャンル混合） | パーソナライズ前提の機能 | Phase 1 |
| マップ可視化 | 記録が溜まらないと価値が出ない | Phase 1 |
| バッジ | コアループの検証が先 | Phase 1 |
| ストリーク | 継続性の前に「1回目の体験」を磨く | Phase 1 |
| 気分選択 | パーソナライズ導入後に検討 | Phase 1 |
| 感想メモ・評価 | コアループに必須ではない | Phase 1 |
| 検索半径設定 | デフォルト3kmで固定 | Phase 1 |

#### Phase 0の画面構成（最小3画面）

| 画面 | パス | 説明 |
|------|------|------|
| ランディングページ | `/` | 1画面LP: 課題→解決→CTA + ログイン/新規登録（MakersGuild 2.4準拠） |
| ホーム兼提案画面 | `/home` | 提案カード + 「行った！」ボタン（**コア画面**） |
| 履歴 | `/history` | 訪問記録リスト |

### 4.3 Phase 1 — コア完成

Phase 0のフィードバックを反映し、以下を追加。Phase 0で後回しにしたゲーミフィケーション・パーソナライズ・OAuth をここで実装する。

#### A. 認証の拡充

| 項目 | 詳細 |
|------|------|
| Google OAuth | Phase 0のメール+パスワード認証に加えてOAuthを追加 |
| プロフィール | 表示名、アバター画像 |

#### B. ゲーミフィケーション（Phase 0から移動 + 強化）

| 項目 | 詳細 |
|------|------|
| XP | 訪問ごとにXPを付与。脱却訪問はボーナス |
| レベル | 総XPに応じたレベルアップ + 称号表示 |
| 進捗表示 | ホーム画面にレベル・XP・次のレベルまでのプログレスバー |
| バッジ | 特定条件で付与（「初脱却」「3ジャンル制覇」等） |
| ストリーク | 週に1回以上訪問すると継続 |
| ジャンル熟練度 | ジャンルごとのXP・レベル管理 |

#### C. 提案のパーソナライズ（Phase 0から移動 + 強化）

| 項目 | 詳細 |
|------|------|
| 興味タグ選択 | オンボーディングで興味タグを3つ以上選択 |
| パーソナライズ提案 | 興味タグに基づき、一定確率で「興味外」のジャンルを混ぜて提案 |
| 脱却モード | 興味ジャンル外の提案を意図的に混合 |
| 気分選択 | 「リラックス」「体を動かしたい」「刺激がほしい」「学びたい」のプリセット |
| 検索半径設定 | 1km / 3km / 5km / 10km から選択 |
| 脱却頻度設定 | 低・中・高 で調整 |

#### D. 記録・可視化の強化

| 項目 | 詳細 |
|------|------|
| マップ表示 | Google Maps上に訪問済みピンを表示 |
| 感想メモ | 一言メモ（任意）と5段階評価 |
| タイムライン | 訪問履歴を時系列で閲覧 |

#### Phase 1の追加画面

| 画面 | パス | 説明 |
|------|------|------|
| ログイン | `/login` | Google OAuthボタン + メール/パスワードログイン |

| マップ画面 | `/map` | 訪問済みピン表示 |
| プロフィール | `/profile` | レベル・バッジ・統計 |
| 設定 | `/settings` | 興味タグ・半径・脱却頻度・アカウント |

### 4.4 Phase 2 — AI強化・通知

#### A. AI提案機能（Gemini API連携）

| 項目 | 詳細 |
|------|------|
| 行動パターン分析 | 訪問履歴からジャンル偏り・時間帯・頻度を分析 |
| パーソナライズ提案 | 「あなたはカフェが多いので、今日は美術館はいかが？」等の文脈ある提案 |
| 気分 × 履歴の掛け合わせ | 選択した気分と過去の行動パターンを組み合わせた提案生成 |
| 提案理由の説明 | なぜこの場所を提案したかをAIが自然言語で説明 |

#### B. 通知・リマインダー

| 項目 | 詳細 |
|------|------|
| ストリークリマインダー | ストリーク切れが近い場合に通知 |
| 週次サマリー | 週の訪問数・獲得XP・新ジャンル挑戦をメールで通知 |
| おすすめ通知 | 天気や曜日に合わせた提案プッシュ通知（Web Push API） |

### 4.5 Phase 3 — ソーシャル・プレミアム

#### A. ソーシャル機能

| 項目 | 詳細 |
|------|------|
| フレンド | ユーザー同士のフレンド登録 |
| ランキング | 週間XPランキング（Duolingoのリーグ制を参考） |
| 訪問共有 | 訪問記録をフレンドにシェア |
| グループチャレンジ | 友人同士で「今週3箇所新しい場所に行く」等のチャレンジ設定 |

#### B. フリーミアムモデル（MakersGuild 3.1準拠）

| プラン | 月額 | 内容 |
|--------|------|------|
| **Free** | ¥0 | 提案（1日3回）、訪問記録、基本XP/レベル、バッジ、ストリーク |
| **Premium** | ¥580/月 (年額 ¥4,800) | AI提案無制限、詳細な行動分析レポート、プレミアムバッジ、高度なフィルター |

**Freeの設計方針（MakersGuild 3.1準拠）**:
- 無料でも十分にコアループ（提案→訪問→XP）を体験可能
- 「もっと使いたい」と思った時点で自然にPremiumを提示
- 1日3回の提案制限がコンバージョンポイント

**価格根拠（MakersGuild 3.2参考）**:
- ターゲット層の支払い意思: 月500〜1,000円
- 年額プランで実質月400円にディスカウント → 「缶コーヒー1本分」の心理的アンカー
- 初期100ユーザー有料化 × ¥580/月 = MRR ¥58,000（Google Maps API費用をカバー可能）

| 項目 | 詳細 |
|------|------|
| 技術選定 | React Native |

## 11. iOS化に向けた考慮事項

### 11.1 Web版での準備

| 項目 | 対応 |
|------|------|
| API設計 | RESTful APIをWeb/iOS両対応で設計 |
| 認証 | OAuth + JWT方式でプラットフォーム非依存 |

| レスポンシブ | モバイルファーストでUI設計（iOS化時の操作感を事前検証） |

### 11.2 技術選定の判断基準（後日決定）

| 選択肢 | メリット | デメリット |
|--------|---------|----------|
| **React Native** | Reactの知見を活用可能。Web版コードの一部再利用 | ネイティブ機能のブリッジが必要な場合がある |
| **Swift (ネイティブ)** | 最高のパフォーマンス・UX。Apple APIとの親和性 | Web版と別コードベース。開発コスト高 |
| **Flutter** | クロスプラットフォーム対応。高パフォーマンス | Dart習得が必要。Reactの知見が活かせない |
| **PWA** | 追加開発不要。Web版をそのまま配布可能 | プッシュ通知やバックグラウンド処理に制限あり |

### 11.3 iOS固有の要件（将来）

- Apple Sign-In対応（Google OAuthを提供する場合、App Storeガイドライン4.8により必要）
- App Tracking Transparency対応
- バックグラウンド位置情報の適切なハンドリング
- App Store審査ガイドラインへの準拠
- アプリ内課金（In-App Purchase）でのプレミアムプラン提供

## 12. リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| Google Maps API費用増大 | コスト増 | Places APIレスポンスをRedisキャッシュ（TTL 24h）。1日あたりのリクエスト上限を設定 |
| 提案の質が低い | ユーザー離脱 | Phase 0でドッグフーディングし、Phase 1でフィードバックを集めてロジックを反復改善 |
| 「1回で飽きる」問題 | リテンション低下 | XP/レベルの即時フィードバックで「もう1回」を誘発。ストリーク（Phase 1）で週次習慣化 |
| Gemini API の応答遅延 | UX低下 | タイムアウト設定 + フォールバック（ルールベース提案） |
| 位置情報の精度不足 | 提案品質低下 | ユーザーによる手動位置調整を許可 |
| 個人情報漏洩 | 信用失墜 | 最小限のデータ保持。精密位置データは端末のみに保持 |
