# Roamble — 要件定義書

> 新しいお店・場所になかなか行けない人のためのWebアプリ
> 最終更新: 2026-03-11（フェーズ順序修正: Phase 2=通知、Phase 3=iOS、Phase 4=AI強化、Phase 5=ソーシャル）

---

## 1. 機能要件

### 1.1 フェーズ分け（MakersGuild MVP思考準拠）

| フェーズ | 名称 | 方針 | 主な機能 |
|---------|------|------|---------|
| **Phase 0** | 最小公開 | Build in Publicに留めて本番環境公開しない。少しでもフィードバック獲得しながら「セキュリティ的に出しても問題ないもの」を作るまで耐える | 提案 + 訪問記録 + 簡易認証（最低限） |
| **Phase 1** | コア完成 | Phase 0のフィードバックを反映して磨き上げ。本番公開 | Google OAuth + XP/レベル/ゲーミフィケーション + 興味タグ/パーソナライズ + バッジ + ストリーク + マップ可視化 |
| **Phase 2** | 通知機能 | リテンション向上のためWeb Push・メール通知を実装。設定画面で各通知の有効/無効を制御可能 | Web Push（PWA）+ メール通知 + 通知設定UI |
| **Phase 3** | iOSネイティブ | Web β版のMVPをiOSアプリとして移植。フリーミアムモデルも同時導入 | React Native移植 + フリーミアムモデル（Free/Premium）|
| **Phase 4** | AI強化 | ユーザーが定着したらAIを強化 | Gemini AI提案 + パーソナライズ高度化 |
| **Phase 5** | ソーシャル | 成長フェーズ | フレンド + ランキング |

### 1.2 Phase 0 — 最小公開（Minimum Viable Launch）

> **MakersGuildの原則**: 「少し恥ずかしい」程度（完成度70%前後）で公開するのがちょうど良いタイミング。機能は1〜2本線で十分。フィードバックが一点集中して鋭くなる。

#### コア機能（これだけ作る）

**A. 提案機能**

| 項目 | 詳細 |
|------|------|
| 位置情報取得 | ブラウザのGeolocation APIで現在地を取得。許可が拒否された場合はモーダルを表示し、ユーザーが「設定で許可する」または「渋谷駅周辺で試す」を選択できる。後者を選んだ場合、デフォルト位置（渋谷駅周辺）で提案を生成し、トーストで「現在地を取得できないため、渋谷駅周辺で表示しています」と通知する。また AppHeader の位置アイコン（`location_on`）を黄色の `location_off` に切り替えることで、別画面から戻った際もデフォルト位置使用中であることを視覚的に示す |
| 施設検索 | Google Maps Places APIで周辺施設を検索 |
| 提案ロジック | 現在地周辺の施設をカード表示。パーソナライズなし（ジャンルフィルター程度） |
| 提案表示 | カード形式（施設名・写真・ジャンル・距離）。1枚ずつ表示 |
| 1日の提案枚数 | **1日3枚まで**。ページ訪問時にその日の提案カード3枚を生成（API1回で3件取得） |
| カードループ | 3枚目をスキップすると1枚目に戻る（3枚のカードを循環表示） |
| スキップ | スキップボタンまたは左スワイプで次のカードへ進む |
| 訪問記録アクション | 各カードに「行ってきた！」ボタンを配置。どのカードからでも訪問記録可能 |
| 除外ロジック | 訪問済みの場所は提案候補から除外 |
| 日次リセット | 0:00（JST）に提案カードがリセットされ、翌日は新しい3枚が生成される |

**B. 訪問記録（最低限）**

| 項目 | 詳細 |
|------|------|
| 記録 | 「行った！」ボタンで訪問を記録 |
| 履歴 | 訪問履歴の一覧表示（リスト形式） |

**C. 認証（最低限）**

| 項目 | 詳細 |
|------|------|
| 認証方式 | ~~メール+パスワード認証（JWT）~~ → **Google OAuth認証のみ**（実装済み） |
| パスワード保存 | ~~bcryptでハッシュ化~~ → 不要（Google管理） |

- 1日3枚のカードが生成される（API1回のリクエストで3件取得）

### 1.3 Phase 1 — コア完成

Phase 0のフィードバックを反映し、以下を追加。Phase 0で後回しにしたゲーミフィケーション・パーソナライズ・OAuth をここで実装する。

#### A. 認証の移行

| 項目 | 詳細 |
|------|------|
| Google OAuth | ~~Phase 0のメール+パスワード認証に加えてOAuthを追加~~ → **メール+パスワード認証を廃止し、Google OAuth認証のみに統一** |
| プロフィール | 表示名（変更可）、アバター画像（Googleプロフィール画像を自動使用） |

#### B. ゲーミフィケーション（Phase 0から移動 + 強化）

| 項目 | 詳細 |
|------|------|
| XP | 訪問ごとにXPを付与。チャレンジ訪問はボーナス |
| レベル | 総XPに応じたレベルアップ + 称号表示 |
| 進捗表示 | プロフィール画面にレベル・XP・次のレベルまでのプログレスバー |
| XP獲得演出 | 「行ってきた」ボタンを押した時にマップにピンが刺さってXP獲得モーダルを表示 |
| バッジ | 特定条件で付与（「初チャレンジ」「3ジャンル制覇」等）。獲得時はトースト通知 |
| ストリーク | 前回訪問から7日以内に訪問すると継続（ローリングウィンドウ方式）。8日以上空くとリセット |
| ジャンル熟練度 | ジャンルごとのXP・レベル管理（最大Lv.30、ユーザーレベルと同じ指数カーブ閾値）。訪問で獲得したXP（ボーナス含む）がジャンル熟練度にも同額加算される |

#### C. 提案のパーソナライズ（Phase 0から移動 + 強化）

| 項目 | 詳細 |
|------|------|
| 興味タグ選択 | オンボーディングで興味タグを3つ以上選択 |
| パーソナライズ提案 | 興味タグに基づきパーソナライズ提案。強制的な興味外挿入はなし（熟練度ベースで自然にチャレンジが発生する設計）|
| チャレンジモード | **興味タグ外 × 熟練度ベース判定**: 「興味タグ外 かつ ジャンル熟練度Lv.5以下」のジャンルへの訪問をチャレンジ扱い（+100XP）。興味タグ内のジャンルへの訪問はチャレンジ扱いしない。ジャンル熟練度の最大はLv.30なので、興味タグ外でもLv.6以上になれば通常扱い |
| 検索半径設定 | 3km〜30km のスライダーで選択（デフォルト10km）|
| ~~チャレンジ頻度設定~~ | ~~低・中・高 で調整~~ → **廃止**（強制挿入ロジックを削除。3件全てをランダム/パーソナライズで提案し、興味タグ外のジャンルで自然にチャレンジが発生する設計に変更）|
| 提案リロード | ホーム画面のリロードボタンで当日の提案を再取得できる。**Web β版: 1日3回まで**。iOS版: Free=1回/日、Premium=3回/日。残り回数をUIに表示 |
| 設定変更の即時反映 | 興味タグ・提案半径の変更時にリロード1回分を消費してキャッシュをクリアし、新設定で提案を即時更新できる。リロード残0の場合はキャッシュクリアなしで設定のみ保存し、翌日リセット時に反映（確認モーダルでユーザーに通知。クエリパラメータ `refresh_suggestions=true` でバックエンドがキャッシュクリア + リロードカウントをインクリメント） |
| 訪問ボタンの有効条件 | 現在地と施設の距離が **200m以内** の場合のみ「行ってきた！」ボタンを有効化。距離が遠い場合はボタンをグレーアウトし「この場所に到着してから記録できます」と表示。`ENVIRONMENT=development` の場合は距離制限なしで常に有効（開発・テスト用）。**バックエンドでも二重検証**: `POST /api/visits` のリクエストボディに `user_lat`, `user_lng` を含め、バックエンドでHaversine距離計算を行う。200m超でかつGPS取得済み（0,0でない）の場合は `400 Bad Request {"code": "TOO_FAR_FROM_PLACE"}` を返す。`ENVIRONMENT=development` の場合はバックエンド検証もスキップ。 |

#### D. 記録・可視化の強化

| 項目 | 詳細 |
|------|------|
| マップ表示 | Google Maps上に訪問済みピンを表示。マップ初期表示時の中心位置はユーザーの現在地（Geolocation API）を基準とする。現在地が取得できない場合は訪問履歴の座標平均を使用する |
| 訪問詳細画面 | 履歴画面の各アイテムをタップすると詳細画面に遷移。一言メモ（任意）と5段階評価を入力・表示 |
| タイムライン | 訪問履歴を時系列で閲覧（一覧表示は現在のまま維持） |
| Google Maps で開く | 提案カードから施設の Google Maps ページを開くリンクを表示（`https://www.google.com/maps/place/?q=place_id:{id}`）。ナビゲーション起動・施設詳細確認に活用できる。Issue #200、Phase 1 ファイナル実装済み |
| 営業時間表示 | 提案 API が Places API の `currentOpeningHours` を取得し、現在営業中か否か（`open_now`）をレスポンスに含める。フロントエンドで営業状況をカード上に表示。Issue #201、Phase 1 ファイナル実装済み |

#### E. 公開ページ・運用機能

| 項目 | 詳細 |
|------|------|
| ベータ合言葉アクセス制限 | `VITE_BETA_PASSPHRASE` 環境変数で設定した合言葉を入力したユーザーのみ `/home` へアクセスできる。規定外のユーザーは `root.tsx` clientLoader で `/beta-gate` にリダイレクトされる。ベータ期間終了後は本変数を削除し beta-gate ルートを廃止 |
| GA4 アナリティクス | `VITE_GA4_ID` 環境変数で GA4 測定 ID を設定。ページビュー・訪問記録・バッジ獲得・チャレンジ訪問などのイベントを送信。未設定時は noop（ローカル開発では送信されない） |
| PWA 対応 | `vite-plugin-pwa` による Service Worker + Web App Manifest。ホーム画面にインストール可能。インストール後は `display: standalone` でアプリライクな UI で動作。インストール促進のための `/pwa-prompt` 画面あり |
| OGP / SNS シェア | `og:title` / `og:image` / Twitter Card を `vite.config.ts` の `injectOgpPlugin` でビルド時に静的注入。OGP メイン画像は `https://roamble.app/ogp.png` |

### 1.4 Phase 2 — 通知機能

> Web Push通知（PWA）とメール通知を実装し、ユーザーのリテンション向上を図る。設定画面から各通知の有効/無効を制御できる。詳細は `docs/notification-roadmap.md` を参照。

#### 通知種別

| 通知 | Web Push | Email | タイミング | 送信条件 |
|------|:--------:|:-----:|-----------|---------|
| 毎日提案リフレッシュ | ✅ | ❌ | 毎日 7:00 JST | Push購読済み・ON |
| ストリークリマインダー | ✅ | ✅ | 毎週日曜 7:00 JST | ストリーク > 0 かつ今週未訪問 |
| 週次サマリー | ✅ | ✅ | 毎週月曜 10:00 JST | 設定ON（訪問なしでも必ず送る） |
| 月次サマリー | ✅ | ✅ | 毎月1日 10:00 JST | 設定ON（訪問なしでも必ず送る） |

#### 技術構成

| 要素 | 詳細 |
|------|------|
| Web Push | `webpush-go`（VAPID方式）+ PWA Service Worker（`injectManifest` モード） |
| メール | Resend（`notifications@roamble.app`）|
| スケジューラー | `robfig/cron v3`（Goプロセス内、`Asia/Tokyo` タイムゾーン）|

### 2.5 既知の制約

#### 複数端末同時利用（Phase 1 時点）

- **現状**: JWT 方式のため、複数端末でのログインは技術的に可能。ただしフロントエンドキャッシュ（React Query）は端末ごとに独立しているため、端末間でのリアルタイム同期は保証されない。
- **具体的な問題例**: 端末 A で訪問記録を行っても、端末 B の XP・履歴表示には即座に反映されない（次回ページ遷移 or 手動リロードで解消）。
- **推奨運用**: 1 ユーザー 1 端末での利用を想定した設計。同一アカウントの複数端末同時利用はデータの不整合が発生する可能性がある。
- **根本対応**: iOS ネイティブ版（Phase 3）にて WebSocket またはプッシュ通知による同期を検討。

| リスク | 影響 | 対策 |
