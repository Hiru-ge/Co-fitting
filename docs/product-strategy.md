## 2. プロダクト概要

### 2.1 ビジョン

**「また同じお店になってしまう人の、最初の一軒を変える」**

Roambleは、毎回同じ店に落ち着いてしまう人が「今日は違うお店に入れた」という成功体験を積み重ねることを支援するアプリケーションである。ランダム提案で選ぶ手間をなくし、ゲーミフィケーション（XP・レベル・バッジ）で「入れた」を継続可能な習慣にする。

### 2.2 解決する課題

> 「新しいお店に行きたいけど、知らない店に一人で入るのが怖い。結局いつもの店に落ち着いてしまう」

- **課題の頻度**: 毎週末（週1〜2回）発生する日常的な課題
- **課題の深さ**: 「知らない店への心理的ハードル」は行動を実際に制限しており、慢性的な停滞感として蓄積する。単なる「めんどくさい」より深い
- **課題の二層構造**: (1) 心理的障壁 — 知らない店に一人で入るのが怖い (2) 選択コスト — 自分で探すと情報過多で疲れる。この2つが重なり「いつもの店」に収束する
- **既存の解決策の不足**: 食べログ・Retty・Googleマップは「行きたいお店が決まっている人」または「能動的に探せる人」向け。「何が良いかわからない、しかも入るのが怖い」人には機能しない

### 2.3 アイデア3条件スコアリング（MakersGuild 1.2準拠）

| 条件 | 評価 | 根拠 |
|------|------|------|
| **課題感** | ★★★☆☆ | 「死ぬほど困る」ではないが、行動を実際に制限している慢性的な課題。「怖い」は「めんどくさい」より課題感が深い。ゲーミフィケーションで成功体験を積み重ね、心理的ハードルを下げる |
| **予算** | ★★☆☆☆ | B2C。無料ユーザーが大半になる。フリーミアムで月500〜1,000円程度の課金を狙う |
| **タイミング** | ★★★★☆ | コロナ後の外出回帰トレンド。AI提案の一般化。ゲーミフィケーションアプリの成熟 |

**総合判断**: 課題感と予算が弱めだが、ゲーミフィケーションによる継続性と「楽しさ」で補完する戦略。学習目的も兼ねているため、Go。

### 2.4 コア価値

| 価値 | 説明 |
|------|------|
| **発見** | 普段行かないジャンルの知らないお店との出会いを創出する |
| **開拓の実感** | XP・レベル・バッジで「お店開拓の軌跡」を可視化する。「何軒開拓したか」が積み上がる |
| **適度な背中押し** | 「自分で探す→自分で決める→自分で入る」の心理的負荷を、ランダム提案によって軽減する。「ちょうどいい一歩先」を提案する |

### 2.5 アハモーメント（MakersGuild 3.4準拠）

> **「提案されたお店に"怖いけど入ってみよう"と思えて、実際に入ったら良いお店だった。また来たいと思えた。」**

この体験を**初回利用時に**到達させることが最重要。「怖かったけど入れた」という成功体験が次の行動への自信になる。Phase 0では提案精度に注力し、Phase 1でゲーミフィケーション（XP獲得演出・レベルアップ）を加えて「もう1回」を誘発する。

### 2.6 提案ジャンルスコープ

**提案する対象**: 飲食店・カフェ・バー・スイーツ・ベーカリー・小売店（アパレル・雑貨・書店など）、営業しているあらゆる「お店」

**提案しない対象**: 公園・神社・自然スポット・観光地・アミューズメント施設。「お店に入る」という体験に直結しないものは一切提案しない。

この方針はASOキーワード・LP文言・SNS投稿・提案ロジックにも一貫して反映する。Roambleは「場所探索アプリ」ではなく「お店開拓アプリ」である。

## 3. ターゲット設定（MakersGuild 1.4準拠）

### 3.1 初期ニッチ: ドッグフーディング起点

**将来的には幅広い層を狙うが、MVP段階では自分自身の課題感を起点にターゲットを絞る。**

#### プロトタイプペルソナ（= 開発者自身）

| 項目 | 詳細 |
|------|------|
| 属性 | 都市部在住、20-30代、学生・社会人 |
| 課題 | 新しいお店に行きたいが、知らない店に一人で入るのが怖い。結局いつもの店に落ち着く |
| 行動 | 気になる店の前を素通りしてチェーン店に入る。食べログで探すと情報過多で疲れる |
| 欲求 | 「ここ行ってみたら？」と背中を押してくれる存在がほしい。入れたお店の記録を残したい |
| 支払い意思 | 月500〜1,000円なら出せる |

### 3.2 ニッチ市場評価（MakersGuild 1.4準拠）

| 評価項目 | スコア | 根拠 |
|---------|--------|------|
| 市場の明確さ | 4/5 | 「お店開拓 × ランダム提案 × ゲーミフィケーション」は明確 |
| 競合の少なさ | 4/5 | 「日常圏 × ランダム提案 × ゲーミフィケーション」の組み合わせは未開拓 |
| 課題の深さ | 3/5 | 「怖い」は行動を実際に制限しており、単なるマンネリより深い。ただし生死に関わるレベルではない |
| 到達可能性 | 3/5 | X/Reddit/Product Huntで到達可能だが、ニッチコミュニティが分散 |
| 拡張性 | 4/5 | グルメ特化→旅行先のお店開拓→フィットネス・習慣系など隣接市場への拡張余地あり |

**合計: 18/25** → 有望ライン（15点以上）をクリア

### 3.3 ニッチ → 拡大の戦略

```
Phase 0: 自分自身で検証
    ↓
Phase 1:   都市部の20-30代・お店のマンネリに悩む社会人（Product Hunt / X でリーチ）
    ↓
Phase 2:   Web Push・メール通知 → リテンション強化（アプリを開く強制力を持たせる）
    ↓
Phase 3:   iOSネイティブアプリ移植 + フリーミアムモデル導入
    ↓
Phase 4:   AI提案強化。新生活者（引っ越し直後）、旅行先のお店開拓に拡大
    ↓
Phase 5:   ソーシャル機能。開拓記録を友人とシェアする機能
```

## 4. 競合分析（深掘り版）

### 4.1 競合マップ

| サービス | カテゴリ | 主な機能 | Roambleとの差別化 |
|---------|---------|---------|-------------------|
| **食べログ / Retty** | グルメ情報 | 口コミ・評価から能動的にお店を探す | 「探せる人向け」のツール。「何を食べるか決まらない、入る勇気がない」人には機能しない。Roambleは提案してもらう体験 |
| **Googleマップ** | 地図・検索 | 行きたい場所を能動的に検索・案内 | 情報量が多く選択疲れが起きる。目的地が決まっていない人向けではない |
| **Randonautica** | ランダム探索 | 量子乱数で座標を生成し冒険を促す | ランダム性のみで「パーソナライズ」がない。実在のお店に案内しない。私有地や水域に案内される問題あり |
| **世界の霧 (Fog of World)** | マップ探索ゲーム | 訪問済みの場所の「霧」が晴れる可視化 | 「記録・可視化」に特化し「提案」がない。Roambleは提案→行動→可視化の全ループをカバー |
| **Habitica** | ゲーミフィケーション習慣 | RPG要素でタスク・習慣をゲーム化 | 汎用的な習慣管理でお店に非対応。Roambleはお店開拓特化のゲーミフィケーション |
| **Swarm (Foursquare)** | チェックイン・記録 | 訪問記録・リスト・チェックイン | 記録ツールで「提案」がない。衰退傾向（2024年サービス縮小） |

### 4.2 市場ポジショニング

```
                 提案・発見（受動的）
                     ▲
                     │
        Randonautica │  ★ Roamble
       (完全ランダム) │  (お店特化 × パーソナライズ × ゲーミフィケーション)
                     │
  日常 ◄─────────────┼─────────────► 旅行・観光
                     │
  世界の霧 / Swarm   │  食べログ / Retty / Googleマップ
  (記録・可視化のみ)  │  (能動的検索・口コミ)
                     │
                     ▼
                 検索・能動探索
```

### 4.3 差別化戦略

1. **「日常のお店マンネリ解消」という独自ポジション**: 旅行でもなく、完全ランダムでもない。「知らない店に入るのが怖い」という壁を、提案と報酬で乗り越える
2. **提案→行動→報酬の完結ループ**: 食べログは「探すだけ」、Randonauticaは「座標生成のみ」。Roambleは提案からXP獲得まで一気通貫
3. **ゲーミフィケーション × お店開拓**: 「怖い → 入れた → XP獲得」のループで心理的ハードルを段階的に下げる。「何軒開拓したか」が積み上がる数字が、継続とシェアの動機になる

## 5. 検証計画（MakersGuild「作りながら検証」方式）

> **判断メモ（2026-02-07）**: Makers Guildは開発前の本格的な市場検証を推奨するが、Roambleはドッグフーディング前提・アイデア3条件スコアリング済み（18/25点）・競合分析済み・Phase 0自体が検証フェーズであるため、フルセットの検証は不要と判断。実施したのは「LP+メール収集」「X共感投稿」「Xキーワード検索」の3つのみ。検証の結果、課題は実在することを確認（X検索・LPコピー語彙は `docs/marketing/marketing-strategy.md §8` に記録）。

### 5.1 Phase 0 リリース戦略

Phase 0は**公開環境には置かない**。自分だけでテストし、外部フィードバックはBuild in Public投稿へのコメントに限定する。

| ステップ | 内容 |
|---------|------|
| **Build** | Phase 0のコア機能を実装 |
| **ドッグフーディング** | 自分自身で2週間使い、提案精度とUXを検証 |
| **Build in Public** | X (#BuildInPublic) で開発進捗を共有。コメントから定性フィードバックを得る |

友人テスト・Product Huntローンチ・一般公開はPhase 1以降で実施する。

### 5.2 検証指標

| 指標 | Phase 0 目標 | 判断基準 |
|------|-------------|---------|
| ドッグフーディング継続 | 2週間の自己利用 | 自分が使い続けたいと思えるか |
| 提案→訪問率（自分） | 感覚的に「提案を見て"怖いけど行ってみよう"と思えるか」 | 行動に移せない提案が8割以上なら提案ロジックを見直し |
| Build in Public反応 | いいね・コメントの有無 | 課題共感の言語化データが得られること |

### 5.3 Phase 1以降のローンチ計画（MakersGuild準拠）

**Product Hunt**（Phase 1完成後）:
- 13:01 JST に投稿（日付切替直後）
- タグライン例: "Always ending up at the same restaurant? Roamble picks a new spot for you — one random suggestion at a time, with RPG-style rewards"
- コメント欄をAMA形式で運用。開発背景・ロードマップ・欲しいフィードバックを明示

**X (#BuildInPublic)**（Phase 0から継続）:
- 開発中の進捗を継続的に共有し、ローンチ日にRT部隊として機能してもらう
- ビフォーアフター（いつもの店を素通り → Roambleの提案で新しい一軒に入れた体験）を固定ツイート

**Reddit**（Phase 1完成後）:
- r/SideProject, r/AppIdeas, r/gamification に投稿
- 体験談 + オープン質問形式（MakersGuild推奨フォーマット）

### 5.4 フィードバック収集・管理

| 目的 | ツール |
|------|--------|
| バグ報告 | GitHub Issues |
| 機能要望 | GitHub Discussions or Tally（アンケート） |
| 定性フィードバック | DM / メール → Notionに整理 |

## 7. リテンション戦略（MakersGuild 3.4準拠）

### 7.1 初期体験の最適化

| 施策 | 詳細 | Phase |
|------|------|-------|
| 即時コア体験到達 | 登録後30秒以内に最初の提案を表示 | Phase 0 |
| 空の状態のデザイン | 訪問履歴0件の時「最初の一軒を踏み出そう」と提案画面に誘導 | Phase 0 |
| オンボーディング最適化 | 興味タグ選択 → 即座にパーソナライズされた提案を表示 | Phase 1 |
| 初回成功体験 | 最初の訪問記録で大きなXPボーナス + レベルアップ演出。「怖かったけど入れた」を称える | Phase 1 |

### 7.2 エンゲージメントループ

```
提案を受ける → 実際に入る → XP獲得 → レベルアップ → 新しい提案が気になる → ...
                                ↓
                    「○軒開拓」カウント更新（プロフィール最上部に表示）
                                ↓
                    Receipt of Courage（勇気の証明書）生成 → SNSシェア → 新規流入
                                ↓
                          バッジ獲得（Phase 1）
                                ↓
                          ストリーク継続（Phase 1）
```

**Receipt of Courage（勇気の証明書）** — 最優先バイラル施策

訪問完了時にレシート形式の画像を自動生成し、SNSにシェアできる機能。表示項目: お店名・ジャンル・獲得XP・「○軒目の開拓」カウント・取得バッジ（あれば）。Spotify Wrappedと同じ原理——個人化された成果の可視化はシェアされる。「今日○○カフェに初めて入りました / 23軒目の開拓」という1枚がInstagram StoriesやTikTokで流れることが、最大の獲得チャネルになりうる。お店特化だからこそ機能する施策で、「公園に行った証明書」は誰もシェアしない。

### 7.3 離脱防止（Phase 2〜）

| セグメント | 条件 | 施策 |
|-----------|------|------|
| アクティブ | 毎日 | 毎朝7:00の提案リフレッシュ通知でアプリを開くきっかけを作る |
| 離脱リスク | 前回訪問から6日経過 | ストリークリマインダー通知（Web Push + メール）|
| 週次振り返り | 毎週月曜 | 先週の訪問数・獲得XPをまとめた週次サマリー通知 |
| 休眠 | 30日以上ログインなし | 月次サマリーメール（「先月の開拓を振り返ろう」）|
