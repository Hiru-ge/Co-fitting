## 7. 通知機能のトラッキング欠損 🔴

3/17にリリースした通知機能について、**計測できていないことの方が多い**。以下に欠損項目と影響を整理する。

### 7.1 欠損項目一覧

#### ① Push通知の許可・拒否・スキップが計測されていない

**現状の実装（`PushNotificationBanner.tsx`）:**
```tsx
async function handleAllow() {
  await subscribePush(token);  // ← GA4イベントなし
  dismiss();
}
// 「後で」ボタンの dismiss() にもGA4イベントなし
```

**`lib/push.ts` の `subscribePush()`:**
```ts
if (permission !== "granted") {
  return false;  // ← 拒否・未対応時のGA4イベントなし
}
// 成功時もGA4イベントなし
```

**影響:** Push許可率・拒否率・スキップ率が完全に不明。「/settings に4人到達した」ことはわかるが、そのうち何人が実際にPush通知を有効にしたかが計測できていない。

#### ② バナー表示自体がトラッキングされていない

**現状:** `usePushBannerVisible` でバナーが表示されても GA4イベントなし。

**影響:** 「バナーを見た人数」「バナーを見て許可した人数」「バナーを見て後でを押した人数」のすべてが不明。バナーのコンバージョン率が評価できない。

#### ③ バナーはStandaloneモード（PWAインストール済み）限定

**現状の実装（`use-push-banner-visible.ts`）:**
```ts
const visible =
  isStandalone() &&   // ← PWAとしてインストール済みの場合のみ
  Notification.permission === "default" &&
  !dismissed;
```

**影響:** ブラウザからアクセスしているユーザーにはバナーが一切表示されない。今期の訪問者170人のうち、PWAインストール済みの比率が不明なため、実際に通知許可の機会を得たユーザー数が把握できていない。

#### ④ Push通知クリックによるセッションがDirect扱いになっている

**現状の実装（`public/sw.js`）:**
```js
self.addEventListener("notificationclick", (event) => {
  event.notification.close();
  event.waitUntil(clients.openWindow(event.notification.data?.url || "/"));
  // ↑ UTMパラメータなし
});
```

**影響:** Push通知をクリックしてアプリを開いたセッションが `direct / (none)` として計上される。通知経由のリピート訪問がどれだけあったか判断できない。direct の 78 sessions の中に通知クリックが含まれている可能性があるが、切り分け不可能。

#### ⑤ NotificationTab（Settings画面）からの購読操作がトラッキングされていない

**現状の実装（`NotificationTab.tsx`）:**
```tsx
async function handleSubscribePush() {
  const success = await subscribePush(token);  // ← GA4イベントなし
}
```

**影響:** Settings画面から通知を有効にしたユーザー数が不明。バナー経由 vs Settings経由のどちらが多いかもわからない。

#### ⑥ 通知設定トグルの変更がトラッキングされていない

**現状:** `handleToggle()` → `updateNotificationSettings()` に GA4イベントなし。

**影響:** 「週次サマリー通知をONにしたユーザー数」「Pushをオフにしたユーザー数」等、どの通知項目が使われているかが完全に不明。

#### ⑦ メール告知にUTMパラメータが付いていない

**現状:** 3/17のメール告知からのクリックが `direct` に混入（GA4の流入元に「email」が現れていないことから確認）。

**影響:** メール告知経由のセッション数が不明。直接流入78 sessionsの中にメール経由が含まれているが分離できない。

### 7.2 計測できていること・できていないことの整理

> ✅ = 計測済み　🔧 = 本レポート分析後に対応済み　❌ = 未対応

| 項目 | 計測状況 |
|------|---------|
| /settings ページ到達数 | ✅ 4 users |
| Push通知バナーの表示数 | 🔧 `push_banner_shown` 追加済み |
| Push通知の許可数（バナー） | 🔧 `push_permission_granted` (source="banner") 追加済み |
| Push通知の拒否数（バナー） | 🔧 `push_permission_denied` (source="banner") 追加済み |
| Settings画面でのPush有効化数 | 🔧 `push_permission_granted` (source="settings") 追加済み |
| バナーの「後で」クリック数 | 🔧 `push_banner_dismissed` 追加済み |
| 各通知項目のON/OFF変更 | 🔧 `notification_setting_changed` (field, value) 追加済み |
| 通知クリック経由のセッション数 | 🔧 sw.js に `utm_source=push_notification` 付与済み |
| メール告知クリック数 | ❌ 未対応（次回メール告知時にUTMパラメータを付与すること） |
| PWAインストール率 | ❌ 不明（pwa-prompt 16PV から間接推定のみ） |

### 7.3 対応済みトラッキング（2026-03-26 実装）

| イベント名 | 場所 | 目的 |
|-----------|------|------|
| `push_banner_shown` | `PushNotificationBanner.tsx` | バナー表示数の計測 |

## 8. 課題と推奨アクション

### 8.1 即対応

#### 🔴 Push通知クリック時のUTM付与（`sw.js`）

通知をクリックした際のURLに `?utm_source=push&utm_medium=notification` を追加するだけで、翌日から通知経由セッションの識別が可能になる。コスト小・効果大。

#### 🔴 メール・X投稿へのUTMパラメータ設定

次回のメール告知・SNS投稿から `utm_source=email&utm_campaign=2026-xx` を付与すること。現状では施策効果の評価が不可能。

#### 🔴 Push許可・拒否イベントの計測

`subscribePush()` の成功・失敗時にそれぞれGA4イベントを送信する。許可率が判明すれば、バナーのUIや表示タイミングの改善判断ができる。

### 8.2 中期対応

#### 🟡 ベータゲート撤廃の検討

beta-gate → login の81%離脱は前回（82%）から改善なし。通知機能の効果を最大化するためにも、まず「使ってもらえる人を増やす」ことが先決。

#### 🟡 通知によるリテンション効果の検証（4月以降）

3/17リリースから2〜4週間後のデータで、通知を許可したユーザーと許可していないユーザーのリピート率を比較すること。ただし上記のUTM・イベント計測を先に整備しないと評価できない。

### 8.3 引き続き注視

- **提案スキップ率70%** → 前回から改善なし。パーソナライズ精度の向上が必要
- **Qiita依存の流入** → 次の記事・コンテンツ計画
