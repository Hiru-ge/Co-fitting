> ### 「コンフォートゾーン」という言葉の扱い方針
>
> 「コンフォートゾーン」という概念は内部ドキュメントからも廃止する。
>
> 理由：「コンフォートゾーン脱却」は意識高い系に見え、ターゲット（新しい店に行きたいけど行けない普通の人）との心の距離が広がる。加えて、Roambleをお店特化に絞った以上、地理的・体験的な「行動範囲の拡大」という概念自体が不要になった。外に出る言葉も内部の思考軸も「お店に入れた」「何軒開拓したか」に一本化する。

## 1. 戦略の核

### 1.1 訴求軸：「新しいお店になかなか行けない人へ」に統一

**ターゲットの痛み**: 「新しい店に行きたいのに、結局いつも同じ店になってしまう」

これは抽象的な「コンフォートゾーン脱却」より市場規模が大きく、ゲーミフィケーション（XP・バッジ・ストリーク）との文脈整合性も高い。入口で「あるある」の共感を作り、ゲーミフィケーションで継続させる設計。

**ポジショニング**: 食べログ・Retty・Googleマップとの差別化は「**選ばなくていい（ランダム提案）× 続けるほど面白くなる（ゲーミフィケーション）**」の2点セット。どちらか一方では弱い。

**提案ジャンルのスコープ**: 飲食店・カフェ・バー・スイーツ・小売店など「営業しているお店」に限定する。公園・神社・自然スポット・観光地は提案しない。Roambleは「お店に入る」という体験に特化したアプリであり、「場所を探索するアプリ」ではない。この方針をASO・LP・SNS投稿・提案ロジックに一貫して反映する。

### 1.2 発信の基本スタンス：等身大のドキュメンタリー

無理なキャラ作りや演出をせず、**「新しいお店に行きたいのに踏み出せない開発者が、自作アプリに背中を押されて（渋々）冒険する記録」** をそのままコンテンツにする。

* **構図**: 「自分のために作った機能なのに、自分で困惑している」
* **メッセージ**: 「ロジック上は正しい提案だけど、感情（恐怖）が追いつかない」

この **「開発者 vs 自分の心理的ハードル」** というリアルな戦いをエンタメとして発信する。

## 2. トーン＆マナー

### 「素朴な独り言」 (Candid Soliloquy)

企業の公式アカウントのような「〜です/ます」調は使わず、個人の日記や友人に話すようなトーンで統一する。

#### ✅ 推奨する振る舞い

* **飾らない**: カッコいい部分だけでなく、ビビって店に入れなかった失敗談も隠さず出す。
* **淡々と**: 感情を過剰に盛らず、事実と素直な感想を述べる。
* **弱音はOK**: 「怖い」「めんどくさい」は共感の源泉なので隠さない。

#### 📝 投稿の型（テンプレート）

1. **意図（Dev）**: 「〇〇な体験をしたくて、この機能を足した」
2. **現実（User）**: 「実際に使ってみたら、想像以上にハードだった/予想外の発見があった」
3. **感想（Emotion）**: 「明日からはもっと手加減設定にしようかな…」

## 3. プロダクト連動バイラル施策

アプリの機能自体に「シェアしたくなる（言いたくなる）仕組み」を組み込む。

### 3.1 冒険の証明書 (Receipt of Courage)（最優先バイラル施策）

> **現状**: 訪問完了時の XP 獲得モーダルは実装済み。シェア用の画像生成（Spotify Wrapped 的なビジュアル）は未実装。**お店特化への方針転換に伴い、最優先バイラル施策として実装予定。**

訪問完了時、Spotify Wrappedやレシートのようなデザインで、その日の「開拓」を画像化してシェアできる機能。「公園に行った証明書」は誰もシェアしないが、「初めて入ったカフェ / 23軒目の開拓」は自然にInstagram StoriesやTikTokに投稿される。お店特化だからこそ機能する施策。

* **表示項目（案）**:
  * 施設名 / 写真
  * 獲得XP
  * 獲得バッジ

* **狙い**: 「おしゃれなカフェに行った自慢」ではなく、**「勇気を出した証明」**としてシェアさせる。

### 3.2 希少性の活用 (Scarcity)

「1日3枚の提案を確認しにいく」という体験を、**「今日の3枚、全部脱却モードだったんだけど」** みたいな会話が朝のSNSで発生するように設計する。

* **狙い**: Wordleのように、「今日の3枚、全部**脱却モード**だったんだけど」といった会話を朝のSNSで発生させる。

> **用語メモ**: アプリ内の難易度概念は「脱却モード」（興味タグ外 かつ ジャンル熟練度Lv.5以下への訪問、+100XP）。「ハードモード」という名称は存在しない。

## 4. チャネル別コンテンツ戦略

### 4.0 アカウント戦略：個人アカウント ／ プロダクトアカウントの分離

**原則：「誰向けの情報か」で分ける**

| 投稿種別 | アカウント | 理由 |
|---------|-----------|------|
| 開発ログ・技術的な話・実装の苦労 | **個人** | 開発者コミュニティ・応援消費狙い |
| 共感あるある・UGC紹介 | **プロダクト** | ユーザーが「使っている人の声」を見る場 |
| 新機能リリース・大きなアップデート | **プロダクト** | ユーザーが知るべき情報 |
| バグ修正（ユーザーが影響を受けていたもの） | **プロダクト** | 「修正しました」1文でよい。謝罪文は不要 |
| バグ修正（内部的・ユーザーが気づいていない） | **個人**（か投稿不要） | プロダクトアカウントに出す価値がない |
| 開発ストーリー・感情・進捗 | **個人** | 応援消費・大学院生×個人開発のナラティブ |

**両方に同じ内容を出さない。** どちらをフォローすべきか分からなくなる。

**プロダクトアカウントの運用指針（Duolingo方式）**
- キャラクター的な親しみやすさを持たせる（マスコット ペンギンの言葉口調で投稿してもよい）
- 共感あるある・ユーザーの使用報告紹介が中心
- アップデート情報はシンプルに「〇〇できるようになりました」の1文でよい
- **TestFlight公開1ヶ月前から本格運用開始**。それ以前（現在〜TestFlight前）はプロダクトアカウントへの投稿は行わない

**ベータ期間中（〜TestFlight公開前）の運用方針**
- プロダクトアカウント: 投稿停止。フォロワーがほぼいない状態での投稿は低エンゲージメントとしてアルゴリズムに評価されるだけで資産にならない
- 個人アカウント: iOS開発の苦労話・品質改善の舞台裏を開発ログとして投稿。投稿内に `@roamble_app` をメンションしてプロダクトアカウントへの動線を維持する
- DM営業（Section 9参照）: プロダクトアカウントへの投稿とは別に、泥臭い個別アプローチでユーザーを集める

**プロダクトアカウントのコンテンツ比率（目安）**:
- 共感あるある・UGC紹介: 40%
- 使用実況・成果報告: 30%
- 機能紹介（スクリーンレコーディング30秒以内）: 20%
- アップデート・修正: 10%

### 4.3 note / Qiita — SEO記事（検索流入）

#### 原則：入口は「悩み」、出口は「Roamble」

既存の2本の記事（note・Qiita）は**「Roamble公開しました」型の告知記事**であり、強い課題感を持ったユーザーの検索意図と接続できていない。

「一人で新しい店に入れない」で悩んでいるユーザーは **Roambleを知らない**。そのため「Roambleとは」ではなく「自分の悩みの解決策」を検索している。記事の入口（タイトル・冒頭）が悩みに寄り添っていなければ、SEO経由では流入しない。

```
❌ 告知記事（検索されない）
「Roamble ベータ版を公開しました」
→ Roambleを知らない人は絶対に検索しない

✅ 課題解決記事（検索される）
「一人でカフェに入れない心理と、少しずつ克服する方法」
→ 悩みを持った人が検索し、末尾でRoambleを知る
```

#### 記事構造の型

1. **タイトル**: ユーザーが検索するであろう悩みワードをそのまま使う
2. **冒頭**: 「わかる」と思わせる共感。開発者自身の体験を添える
3. **本文**: 悩みの構造・克服のヒントを誠実に書く（プロダクト売り込みにしない）
4. **末尾**: 「この課題を解決したくてRoambleを作りました」と自然に紹介

#### 推奨タイトルパターン（課題ワード起点）

| タイトル案 | 狙う検索ワード |
|---|---|
| 「一人でカフェに入れない心理と、少しずつ克服する方法」 | 一人で新しい店に入れない / 一人カフェ 怖い |
| 「毎回同じ店になってしまう理由と、新しいお店に踏み出す方法」 | いつも同じ店 / 新しい店 行けない |
| 「いつも同じ場所ばかりになってしまう理由と、行動範囲を広げるヒント」 | いつも同じ場所 / 行動範囲 狭い |

> **既存記事の扱い**: note・Qiitaの2本は告知記事として完結しているので削除不要。追記で末尾に「この記事で書いた課題を解決したくてRoambleを作りました」と加筆し、LPへのリンクを貼るだけでも効果がある。

## 7. ベータテスター施策

### 7.1 方針：報酬ではなく「貢献感」で動かす

ポイントやクレジットのような外発的報酬は使わない。**フィードバックが実際にプロダクトに反映された実感**を提供することで、「自分がこのアプリを作った」という内発的な動機を引き出す（IKEA効果）。

### 7.2 施策：「あなたの声から生まれた機能」の明示

**フィードバックが実装に反映されたとき、以下の2箇所に名前（またはハンドルネーム）を記載する（任意）。**

| 掲載場所 | 内容 |
|---------|------|
| リリースノート | 「◯◯さんのフィードバックから生まれた機能です」と明記 |

* 名前の掲載は**任意**。希望しない場合は匿名扱い。
* 実装に至らなかったフィードバックも、検討済みであれば返信で必ず報告する。

### 7.3 運用フロー

1. ベータテスターからフィードバック受領（GoogleフォームやDMなど）
2. 実装判断（採用 / 保留 / 不採用）
3. **採用した場合**: リリースノートに貢献者名を記載。本人にも個別で報告。

### 7.4 伝え方（X投稿例）

> 「ベータ版を使ってくれた〇〇さんから『スキップした場所を後から見返せるといい』という声をもらって、今日その機能を実装した。自分では気づかなかった。感謝しかない。」

## 9. DM営業（泥臭い個別アプローチ）

個人開発初期段階でSNSのオーガニック流入を待つだけでは規模が作れない。アルゴリズムに頼らず、自分で見込みユーザーを見つけてアプローチする。TestFlight公開前後から本格始動。

**参考ソース**: 戸田大介 (@todayske / bondavi) のDM手法、および https://www.youtube.com/watch?v=WYsjClSkHuk

### 9.1 いいね集め → DM / フォロワーへのDM

**やること**: X（Twitter）で「新しいお店を開拓したい人、いいねしてください」「いつも同じ店になってしまう人いますか？」のような投稿をして、**いいねしてくれた人・フォロワー**に個別にDMを送る。

**DM文面の構造**（戸田大介 @todayske のアプローチを踏襲）:
1. 突然の連絡への謝罪から始める
2. 作ったものを素直に報告する
3. 「なぜ作ったか・なぜ連絡したか」の個人的な動機・熱量を伝える
4. 自動配信ではなく手動で送っていることを明示する（誠実さの担保）
5. ハードルの低いお願いで締める（「試してみてもらえたら嬉しいです」程度）

**DM文面例（Roamble版）**:
> 突然のご連絡、大変失礼いたします。
>
> 実は最近、「いつも同じ店になってしまう」を解決するアプリ（Roamble）を個人で開発しまして。
>
> 私自身、新しいお店に一人で入るのがずっと怖くて、それが嫌でこのアプリを作ったので、同じ気持ちを持っている方に知っていただきたいという気持ちを抑え切れず、直接ご連絡してしまいました。
>
> 自動配信等ではないのですが、突然のご連絡で不快なお気持ちにさせてしまったら、本当にすみません。もしよろしければ、のぞいてみていただけたら嬉しいです！
>
> https://roamble.app

**文面を送る際の注意**:
- 相手のプロフィールや投稿を一言拾うと、テンプレ感が消える（「〇〇のツイート、わかります」など）
- 売り込みの意図を前に出さない。知ってほしいという気持ちだけを伝える

**投稿例**:
> 「新しいお店になかなか行けない」「いつも同じ店になってしまう」って人、いいねしてください。そういう人向けにアプリ作ってます（TestFlight先行招待あり）

### 9.2 キーワード検索 → リプライ・DM

**やること**: X（Twitter）で以下のキーワードで検索し、ペインポイントを呟いている人に対してリプライまたはDMを送る。

**検索キーワード候補**:
- 「いつも同じ店」
- 「また同じ店」
- 「新しい店 行けない」
- 「一人で新しい店」
- 「お店 開拓」
- 「ランチ 迷う」
- 「初見の店 入りにくい」

**リプライ・DMの文面方針**:
- いきなりアプリのリンクを貼らない。まず共感から入る
- 「自分も同じで、それが嫌で作ったアプリがあります」という流れが自然
- 相手のツイートの内容を具体的に拾う（「〇〇の件、わかります」）。テンプレ感が出ると即スルーされる
- 押しつけがましくならないよう、最後は「興味があれば〜」で締める

**注意事項**:
- X APIの制限（無料プランは月800ツイート送信まで）を意識する
- 同じ文面を大量に送るスパム行為は垢BAN対象。必ず文面を相手に合わせて変える
- 返信が来なくても再DM（リマインド）はしない

## 10. SEO運用メモ（2026-04-15）

Issue #341 のLP改善で採用した検索語は、次の5群。

- 店が決められない
- 入るのに勇気がいる
- 初めて入る店 緊張
- お店開拓 / カフェ開拓
- いつもの店に落ち着く

運用意図は、非ブランド語の検索意図とLP本文の体験文脈を一致させること。title/description/OGPは正規コピー固定を維持し、本文見出し側で検索語を自然に受け止める。

実務ルール:

1. 正規コピー（title/description/OGP）は変えない
2. 非ブランド語は見出し・本文で反映する
3. 1セクションに同一語を過密配置しない
4. Search Consoleの非ブランド表示推移を4週間単位で確認する
