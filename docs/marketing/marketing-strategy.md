# Roamble — マーケティング戦略書

> **コンセプト**: "Reluctant Explorer"（気が進まない冒険者）
> **目的**: 開発プロセスとドッグフーディングの記録を通じて、共感と応援（ファン）を獲得する
> **最終更新**: 2026-03-16（ベータテスト期間終了・Phase 3移行反映版）

---

## 1. 戦略の核：ナラティブ（物語）

### 1.1 基本スタンス：等身大のドキュメンタリー

無理なキャラ作りや演出をせず、**「コンフォートゾーンを抜け出したいと願う開発者が、自作アプリに背中を押されて（渋々）冒険する記録」** をそのままコンテンツにする。

### 1.2 共感の構造：アイロニー（皮肉）

読者（未来のユーザー）は、機能そのものではなく、開発者の **「変化しようとする姿勢」** と **「その過程での葛藤」** に共感する。

* **構図**: 「自分のために作った機能なのに、自分で困惑している」
* **メッセージ**: 「ロジック上は正しい提案だけど、感情（恐怖）が追いつかない」

この **「開発者 vs 自分の心理的ハードル」** というリアルな戦いをエンタメとして発信する。

## 2. トーン＆マナー

### 「素朴な独り言」 (Candid Soliloquy)

企業の公式アカウントのような「〜です/ます」調は使わず、個人の日記や友人に話すようなトーンで統一する。

#### ✅ 推奨する振る舞い

* **飾らない**: カッコいい部分だけでなく、ビビって店に入れなかった失敗談も隠さず出す。
* **淡々と**: 感情を過剰に盛らず、事実と素直な感想を述べる。
* **弱音はOK**: 「怖い」「めんどくさい」は共感の源泉なので隠さない。

#### 📝 投稿の型（テンプレート）

1. **意図（Dev）**: 「〇〇な体験をしたくて、この機能を足した」
2. **現実（User）**: 「実際に使ってみたら、想像以上にハードだった/予想外の発見があった」
3. **感想（Emotion）**: 「明日からはもっと手加減設定にしようかな…」

## 3. プロダクト連動バイラル施策

アプリの機能自体に「シェアしたくなる（言いたくなる）仕組み」を組み込む。

### 3.1 冒険の証明書 (Receipt of Courage)（未実装・将来施策）

> **現状**: 訪問完了時の XP 獲得モーダルは Phase 1 で実装済み。シェア用の画像生成（Spotify Wrapped 的なビジュアル）は未実装。Phase 2〜3 以降で検討。

訪問完了時、Spotify Wrappedやレシートのようなデザインで、その日の「冒険」を画像化してシェアできる機能。

* **表示項目（案）**:
  * 施設名 / 写真
  * 獲得XP
  * 獲得バッジ

* **狙い**: 「おしゃれなカフェに行った自慢」ではなく、**「勇気を出した証明」**としてシェアさせる。

### 3.2 希少性の活用 (Scarcity)

「1日3枚の提案を確認しにいく」という体験を、**「今日の3枚、全部脱却モードだったんだけど」** みたいな会話が朝のSNSで発生するように設計する。

* **狙い**: Wordleのように、「今日の3枚、全部**脱却モード**だったんだけど」といった会話を朝のSNSで発生させる。

> **用語メモ**: アプリ内の難易度概念は「脱却モード」（興味タグ外 かつ ジャンル熟練度Lv.5以下への訪問、+100XP）。「ハードモード」という名称は存在しない。

### 4.3 note / Qiita — SEO記事（検索流入）

#### 原則：入口は「悩み」、出口は「Roamble」

既存の2本の記事（note・Qiita）は**「Roamble公開しました」型の告知記事**であり、強い課題感を持ったユーザーの検索意図と接続できていない。

「一人で新しい店に入れない」で悩んでいるユーザーは **Roambleを知らない**。そのため「Roambleとは」ではなく「自分の悩みの解決策」を検索している。記事の入口（タイトル・冒頭）が悩みに寄り添っていなければ、SEO経由では流入しない。

```
❌ 告知記事（検索されない）
「Roamble ベータ版を公開しました」
→ Roambleを知らない人は絶対に検索しない

✅ 課題解決記事（検索される）
「一人でカフェに入れない心理と、少しずつ克服する方法」
→ 悩みを持った人が検索し、末尾でRoambleを知る
```

#### 記事構造の型

1. **タイトル**: ユーザーが検索するであろう悩みワードをそのまま使う
2. **冒頭**: 「わかる」と思わせる共感。開発者自身の体験を添える
3. **本文**: 悩みの構造・克服のヒントを誠実に書く（プロダクト売り込みにしない）
4. **末尾**: 「この課題を解決したくてRoambleを作りました」と自然に紹介

#### 推奨タイトルパターン（課題ワード起点）

| タイトル案 | 狙う検索ワード |
|---|---|
| 「一人でカフェに入れない心理と、少しずつ克服する方法」 | 一人で新しい店に入れない / 一人カフェ 怖い |
| 「コンフォートゾーンから抜け出したい人へ——日常に小さな冒険を仕込む方法」 | コンフォートゾーン 脱出 / 行動範囲 広げたい |
| 「いつも同じ場所ばかりになってしまう理由と、行動範囲を広げるヒント」 | いつも同じ場所 / 行動範囲 狭い |

> **既存記事の扱い**: note・Qiitaの2本は告知記事として完結しているので削除不要。追記で末尾に「この記事で書いた課題を解決したくてRoambleを作りました」と加筆し、LPへのリンクを貼るだけでも効果がある。

## 7. ベータテスター施策

### 7.1 方針：報酬ではなく「貢献感」で動かす

ポイントやクレジットのような外発的報酬は使わない。**フィードバックが実際にプロダクトに反映された実感**を提供することで、「自分がこのアプリを作った」という内発的な動機を引き出す（IKEA効果）。

### 7.2 施策：「あなたの声から生まれた機能」の明示

**フィードバックが実装に反映されたとき、以下の2箇所に名前（またはハンドルネーム）を記載する（任意）。**

| 掲載場所 | 内容 |
|---------|------|
| リリースノート | 「◯◯さんのフィードバックから生まれた機能です」と明記 |

* 名前の掲載は**任意**。希望しない場合は匿名扱い。
* 実装に至らなかったフィードバックも、検討済みであれば返信で必ず報告する。

### 7.3 運用フロー

1. ベータテスターからフィードバック受領（GoogleフォームやDMなど）
2. 実装判断（採用 / 保留 / 不採用）
3. **採用した場合**: リリースノートに貢献者名を記載。本人にも個別で報告。

### 7.4 伝え方（X投稿例）

> 「ベータ版を使ってくれた〇〇さんから『スキップした場所を後から見返せるといい』という声をもらって、今日その機能を実装した。自分では気づかなかった。感謝しかない。」

### 8.2 LP構成案（ローンチ時の参照用）

**キャッチコピー**: 「知らない場所への一歩を、経験値に。」

**サブコピー**: 気になる店の前を素通りしてしまう。いつも同じ道を選んでしまう。Roambleは、そんなあなたの背中をそっと押すWebアプリ。近くにある「まだ知らない場所」を提案して、日常を小さな冒険に変えます。

**課題共感セクションの箇条書き**:
- 気になる店があるのに、一人で入る勇気が出ない
- 知らない道に曲がるのが、なんとなく怖い
- 結局いつものチェーン店、いつもの散歩ルートに落ち着く
- 「新しい場所に行きたい」のに、いざとなると足が動かない
- 行動範囲が思ったより狭いことに気づいて、ちょっとモヤモヤする
